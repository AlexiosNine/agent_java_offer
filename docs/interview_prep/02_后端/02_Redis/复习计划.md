## 三、各题知识点速查

### 题1：缓存击穿与穿透

| 维度 | 击穿 | 穿透 |
|------|------|------|
| key特征 | 热点key | 不存在的key |
| 失效原因 | 过期时间到了 | 数据本身不存在 |
| 治理核心 | 互斥重建（分布式锁） | 三层防线 |

**穿透三层防线**：参数校验 → 布隆过滤器 → 空值缓存（短TTL 30s~120s）

**击穿互斥重建**：SETNX 分布式锁，只有一个线程查库回填，其他等待或快速失败

---

### 题2：Redis Cluster 与一致性哈希

- **哈希槽**：CRC16(key) % 16384，固定 16384 个槽位，查映射表找节点
- **一致性哈希**：哈希环（2^32），顺时针找第一个节点，增减节点只影响相邻区间
- **虚拟节点**：解决一致性哈希数据倾斜，每个物理节点对应多个虚拟节点
- **Redis Cluster 数据倾斜**：重新分配槽位（CLUSTER REBALANCE）

---

### 题3：RDB 与 AOF

| 维度 | RDB | AOF |
|------|-----|-----|
| 方式 | 全量快照（fork + COW） | 追加命令 |
| 文件大小 | 小 | 大（需重写） |
| 恢复速度 | 快 | 慢（重放命令） |
| 数据安全 | 最后快照后数据可能丢失 | everysec 最多丢 1 秒 |

**AOF fsync 策略**：always（不丢）/ everysec（丢1秒，推荐）/ no（OS决定）  
**数据恢复优先级**：AOF 优先于 RDB  
**混合持久化**：`aof-use-rdb-preamble yes`，AOF = RDB快照 + AOF增量

---

### 题4：Redis 为什么快

**四大核心原因**：
1. 纯内存操作
2. 单线程 + I/O 多路复用（epoll）：单线程监听所有连接，哪个就绪处理哪个
3. 高效数据结构（SDS、跳表、ziplist）
4. pipeline 批量操作减少 RTT

**性能下降场景**：慢命令（KEYS *）/ 大key / AOF always / fork耗时 / 网络问题

---

### 题5：Sentinel vs Cluster

| 维度 | Sentinel | Cluster |
|------|----------|---------|
| 解决问题 | 高可用 | 高可用 + 水平扩展 |
| 节点数量 | 通常 3 个哨兵 | 最少 6 个（3主3从） |
| 数据分片 | 无 | 16384 个槽位 |
| 适用场景 | 数据量小、读多写少 | 大数据量、高并发写 |

**迁移路径**：单机 → 主从+Sentinel → Redis Cluster（redis-shake 在线迁移）

---

### 题6：缓存雪崩

**定义**：大量 key 同时过期，请求集中回源 DB

**止血优先级**：
1. 限流 + 熔断（秒级）
2. 降级（分钟级）
3. 手动预热缓存（分钟级）← 关键！
4. 临时扩容 DB 从库（小时级）

**预防三板斧**：
1. TTL 随机抖动（30min + rand(0~5min)）
2. 热点永不过期 + MQ 主动更新
3. 多级缓存（Caffeine 本地缓存 + Redis）
