## 第一题：缓存击穿与缓存穿透

**面试官提问**：
> 请你用 3 分钟时间，解释一下缓存击穿和缓存穿透的区别，以及你在生产环境中会如何分别处理这两种问题？请结合具体场景说明。

**考生回答**：
> 缓存击穿指请求打到 redis 时，redis 中缓存的 key 刚好失效，导致大量请求透过 redis 打到后端数据库。缓存穿透则是请求打到 redis 时，没有缓存对应的 key，导致大量请求透过 redis 打到后端数据库。生产环境中，针对缓存击穿，我们可以在请求 redis 缓存时，增加一个随机抖动，避免大量请求同时打到 redis 中；也可以借助微服务中请求去重和并发合并机制，打到 redis 的请求量。针对热 key 问题，可以通过监控动态识别热 key 更改其过期时间（不断刷新 expire time）。缓存穿透可以根据缓存内容决定策略，例如 redis 中 key 不存在时返回默认值，布隆过滤器在请求 redis 之前先判断是否存在 key，通过请求合并拦截打到 mysql 的请求。

**综合评分**：6.5/10

### A：知识点确认（/5）
- **得分**：3/5
- **掌握**：区分了击穿和穿透的基本定义；提到了随机抖动、请求去重、热key监控、布隆过滤器
- **遗漏**：
  - 击穿核心治理：互斥重建（分布式锁）
  - 穿透三层防线不完整：漏了参数校验（第一层）
  - 空值缓存 + 短TTL 未提及
  - "随机抖动"用错场景（是雪崩的治理手段，不是击穿）

### B：结构化表达（/5）
- **得分**：3.5/5
- **亮点**：分别给出了两种问题的治理方案；提到了多种技术手段
- **问题**：缺结构化框架；完全缺场景举例；技术手段用错场景

**复盘要点**：
1. 击穿核心是互斥重建（分布式锁），不是随机抖动
2. 穿透三层防线：参数校验 → 布隆过滤器 → 空值缓存（短TTL）
3. 必须给场景：秒杀商品（击穿）、恶意查询不存在ID（穿透）
4. 说清楚本质区别：热点key vs 不存在key

## 第二题：Redis Cluster 与一致性哈希

**面试官提问**：
> 请你用 3 分钟时间，解释一下 Redis Cluster 是如何实现数据分片的？重点说明一致性哈希算法的原理，以及它是如何解决节点增减时的数据迁移问题的。如果遇到数据倾斜，你会怎么处理？

**考生回答**：
> Redis Cluster 通过 CRC 算法计算对应的槽位，将整个整数空间划分成多个槽位，再通过 CRC 算法映射到相应的槽位，同时将整个槽位根据 Redis Cluster 的 node 做划分，每个 node 拥有一部分槽位，当 key 落在某一个槽位，就被存储到对应的 redis 节点中。采取这样的 Redis 槽位算法，主要是为了一致性哈希。这样在后续增减 Redis 节点的过程中，哈希值不变，可以通过手动分配槽位，尽量减少数据转移耗时过长和分布不均的问题。

**综合评分**：6/10（经追问讨论后认可理解方向正确）

### A：知识点确认（/5）
- **得分**：3/5
- **掌握**：Redis Cluster 槽位机制；CRC16 算法；槽位分配到不同节点；增减节点涉及数据迁移
- **遗漏**：
  - 槽位数量：16384 个
  - 一致性哈希原理（哈希环、顺时针查找、虚拟节点）
  - 数据倾斜处理（题目明确要求）
  - Redis Cluster 和一致性哈希的关系（设计理念 vs 具体实现）

### B：结构化表达（/5）
- **得分**：3/5
- **亮点**：提到了槽位机制的基本流程
- **问题**：表达混乱；漏答数据倾斜处理；缺场景举例

**复盘要点**：
1. Redis Cluster 用哈希槽（16384个固定槽位），一致性哈希是理论算法
2. 一致性哈希四要素：哈希环、节点定位、数据定位、增减节点优势
3. 数据倾斜：一致性哈希用虚拟节点；Redis Cluster 用槽位重新分配
4. 题目问什么答什么，数据倾斜处理不能漏

## 第三题：RDB 与 AOF 持久化

**面试官提问**：
> Redis 有 RDB 和 AOF 两种持久化方式。请你用 3 分钟时间，对比这两种方式的优缺点，并说明在什么场景下你会选择哪种方式？如果要同时保证性能和数据安全，你会怎么配置？

**考生回答**：
> RDB 是快照性质的数据存储，用于保存某一时刻的数据快照。AOF 是 append only 的数据存储，由一段时间内的所有命令组合构成。RDB 因为是某一时刻的快照，因此生成的耗时更长，保存的数据更精简，有利于从某个时间点开始读取全量数据快照快速恢复。AOF 因为是某一时间段内的命令集合，因此生成的耗时较短，但是可能有大量重复或者过时的命令，因此针对过大的 AOF 也会有重写 AOF，按照当前 key 的状态重写 AOF 文件，主要用于增量记录和修复。RDB 和 AOF 都可以配置同步生成或者 bg 通过后台进程生成，一般在生产环境下，对实时性要求较高，写入比较频繁的数据一般采用 AOF 持久化，对于实时性要求较低，写入较少的情况下，一般采取 RDB 持久化。但是实际中，一般采用 RDB + AOF 结合的机制，RDB 保存数据快照，AOF 记录增量更新。AOF 同时还提供了三种保存机制，一个是定时保存，一个是每执行一个命令保存一次，还有就是不保存 AOF，现网为了兼顾数据安全和性能，一般采用定时保存 every sec。

**综合评分**：8/10（经追问讨论后调整）

### A：知识点确认（/5）
- **得分**：4/5
- **掌握**：RDB/AOF 基本定义；AOF 重写；后台进程；RDB+AOF 结合；AOF 三种 fsync 策略
- **遗漏**：
  - RDB 的 fork + COW 机制
  - 数据恢复优先级：AOF 优先于 RDB
  - everysec 量化：最多丢 1 秒数据
  - Redis 4.0+ 混合持久化（`aof-use-rdb-preamble yes`，AOF 文件内部混合格式）

### B：结构化表达（/5）
- **得分**：4/5
- **亮点**：有清晰对比框架；有场景选择；有具体配置建议（everysec）
- **问题**：RDB 耗时原因表述不够准确；缺具体场景举例

**复盘要点**：
1. RDB 耗时长的原因：全量写入（fork + COW），不只是"快照性质"
2. everysec 要量化：最多丢 1 秒数据
3. 数据恢复优先级：AOF 优先于 RDB
4. Redis 4.0+ 混合持久化：AOF 文件 = RDB快照 + AOF增量，恢复更快

## 第四题：Redis 为什么快

**面试官提问**：
> Redis 单机可以达到 10 万+ QPS，请你用 3 分钟时间，解释一下 Redis 为什么这么快？重点说明：为什么 Redis 使用单线程模型还能达到这么高的性能？以及在什么场景下 Redis 的性能会下降？

**考生回答**：
> 1. 纯内存，redis 的数据写入与读取都基于纯内存操作，相较于磁盘读取，内存的读写都快了一个数量级。2. 主流程单进程；redis 主流程执行命令是基于单进程设计的，规避了大量的并发问题，因此相较于关系型数据库少了大量的复杂逻辑。3. 不支持完整的事务：redis 作为一个基于内存的缓存，不完全支持关系型数据库中的事务，因此写入和读取时的逻辑更简单，耗时更少。4. redis 的数据结构设计：redis 中设计了多种数据结构，并且基于 C 语言的系统调用执行相应操作，降低操作耗时以及算法的时间复杂度。5. redis 的优化：包括 pipeline，支持 lua 脚本，redis client 降低了多个命令分步执行时的数据传输耗时。
>
> （追问补充）性能下降排查：1. 负载问题：大key（内存占用高）和热key（CPU高、请求倾斜）。2. 性能消耗：AOF always、fork 耗时（结合节点内存辅助判断）。3. 网络问题：网络抖动、机房网络配置变更（真实案例：腾讯云规则变更导致内网→外网→内网路由）。

**综合评分**：7.5/10（含追问补充后）

### A：知识点确认（/5）
- **得分**：3.5/5
- **掌握**：纯内存；单线程规避并发；数据结构设计；pipeline；大key/热key区分；fork耗时；网络问题
- **遗漏**：
  - I/O 多路复用（epoll）：单线程快的核心原因
  - 慢命令（KEYS *、SMEMBERS 等 O(N) 命令）
  - "不支持完整事务"不是 Redis 快的原因（不准确）

### B：结构化表达（/5）
- **得分**：4/5
- **亮点**：有编号列表；追问补充有真实案例（加分）；大key/热key区分维度实用
- **问题**：漏掉 I/O 多路复用；性能下降场景需追问才补充

**复盘要点**：
1. 单线程快的核心：I/O 多路复用（epoll），不是"规避并发"
2. 性能下降场景主动说：慢命令、大key、AOF always、fork耗时、网络问题
3. 去掉"不支持完整事务"这个点，不准确

## 第五题：Sentinel vs Redis Cluster

**面试官提问**：
> 如果你现在要设计一个高可用 Redis 架构，团队里有人建议用 Sentinel，有人建议直接上 Redis Cluster。请你说明：1) Sentinel 和 Cluster 的核心区别是什么？2) 各自适用什么场景？3) 如果业务从单机 Redis 演进到高并发阶段，你的迁移路径会怎么设计？

**考生回答**：
> Sentinel 主要依靠哨兵节点来维持整个 redis 集群的可用性，对于哨兵节点的要求比较高，而且哨兵节点通常不参与 redis 集群中业务数据的存储，对于资源有一定的浪费。当多个哨兵节点报错时，需要重新选举哨兵，并且各个决策需要哨兵中的多数确认才能执行，故障恢复的时间较久。Cluster 则是一个对等的集群，每个 redis 节点通过主从节点来做冗余容错，能够充分利用到每一个 redis 实例的资源，当节点报错时，由从节点提升为主节点即可暂时容错。Sentinel 适用于对故障恢复时间不敏感，redis 主从架构下的高可用。Cluster 适用于对故障恢复时间敏感，大数据量，高并发场景下的高可用，通常是需要水平扩展的 redis 集群的状况下。从单机 redis 演进到高并发阶段，首先当单机到达瓶颈后，先尝试通过主从结构，通过从节点分担读的压力，接着如果数据量同时扩展起来后，会逐渐迁移成 redis cluster，通过水平扩展将数据分片保存在不同的 redis 主从上。

**综合评分**：7/10

### A：知识点确认（/5）
- **得分**：3.5/5
- **掌握**：Sentinel 哨兵机制、多数确认；Cluster 对等架构；场景选择；迁移路径方向
- **遗漏**：
  - Sentinel 描述有偏差："哨兵节点要求高/资源浪费"不准确，哨兵很轻量
  - Sentinel 核心价值：自动故障转移（主节点宕机后自动选主）
  - 关键数字：Sentinel 通常 3 个节点；Cluster 最少 6 个节点（3主3从）
  - 迁移方案：redis-shake 在线迁移，不停服

### B：结构化表达（/5）
- **得分**：3.5/5
- **亮点**：三个子问题都回答了；有清晰对比框架；迁移路径有层次
- **问题**：Sentinel 描述不准确；迁移路径缺数据迁移方案

**复盘要点**：
1. Sentinel 核心是自动故障转移，哨兵节点很轻量
2. 关键数字：Sentinel 3个节点；Cluster 最少 6个节点（3主3从）
3. 迁移路径：单机 → 主从+Sentinel → Cluster，用 redis-shake 在线迁移

## 第六题：缓存雪崩

**面试官提问**：
> 双十一大促期间，你负责的 Redis 集群在零点整出现了大规模缓存失效，导致大量请求直接打到数据库，数据库 CPU 飙到 100%，服务开始超时。请你说明：1) 这是什么问题，根本原因是什么？2) 此刻你会怎么止血？3) 事后你会做哪些预防措施？

**考生回答**：
> 缓存雪崩一般指的是 redis 中有大量的 key 同时过期，导致大量请求直接打到了对应的数据库中。双十一大促作为一个线上电商促销活动，很容易出现零点大量缓存同时过期的情况，根本原因在于设置缓存过期时间时，统一采用了相同的过期时间配置，结合零点有大量请求同时写入 redis 缓存导致了该情况出现。针对线上已经发生了这种情况，首先是进行数据库的扩容，然后将数据库的请求分流（从库），分库分表来降低数据库的负载，接着采用降级策略，将部分非主流程的请求引导到缓存或者说默认返回，降低数据库负载，尽量保护主流程的用户体验；再判断当前流程是否可以异步，通过 Kafka 削峰，用户侧是否可以通过限流直接拦截部分请求，例如直接显示当前接口忙，稍后重试；如果以上都暂时无法缓解，最后则是采用熔断策略，避免更多请求打到数据库，导致情况进一步恶化。事后，首先 redis 配置中在写入时，增加随机抖动，避免大量数据同时过期；其次是设置对应的降级策略和限流策略，尽量在出现同类问题时，尽可能保障用户的体验；最后是完善监控以及拨测压测的系统，尽量做到提前预警。

**综合评分**：7.5/10

### A：知识点确认（/5）
- **得分**：3.5/5
- **掌握**：缓存雪崩定义准确；根本原因分析到位；止血策略多层次；Kafka 削峰；随机抖动；监控
- **遗漏**：
  - 止血优先级有问题：扩容和分库分表不是紧急止血手段
  - 缺少手动预热缓存（止血关键步骤）
  - 预防缺少热点永不过期 + 主动更新
  - 预防缺少多级缓存（Caffeine 本地缓存）

### B：结构化表达（/5）
- **得分**：4/5
- **亮点**：三个子问题都回答了；止血策略有层次；有具体技术手段（Kafka削峰）
- **问题**：止血优先级排序有问题；缺具体数字

**复盘要点**：
1. 止血优先级：限流熔断（秒级）→ 降级（分钟级）→ 预热缓存（分钟级）→ 扩容（小时级）
2. 预热缓存是止血关键：让缓存恢复才是根本解法
3. 预防三板斧：随机抖动 + 热点永不过期 + 多级缓存（Caffeine）
4. 扩容不是紧急手段，先限流再说
