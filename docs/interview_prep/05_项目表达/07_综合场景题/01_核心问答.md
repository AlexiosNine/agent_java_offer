### 原文内容

场景题:

设计一个抢红包的系统，1、规则同wx红包，2、可以设置红包的权限，3、考虑数据量+并发

Gateway -> 红包服务(无状态) -> Redis Cluster(高并发原子扣减) -> MQ -> 账务服务 -> MySQL(主存)

- Redis负责“抢红包瞬时并发”
- MQ解耦削峰，避免直接打库
- MySQL负责“最终账务与审计”



## 补充：如果让你从 0 到 1 搭一个 AI ToC 后端服务，你会怎么拆

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容

1. **接入层**  
API Gateway + 鉴权（JWT）+ 限流（用户/IP/设备）+ 灰度路由。  
长响应用 SSE/WebSocket，短请求走 HTTP。

2. **编排层（Agent Runtime）**  
把流程做成状态机/图：意图识别 -> 检索/工具 -> 生成 -> 审核 -> 返回。  
每个节点有超时、重试、最大步数、熔断和降级。

3. **模型层**  
模型路由（大模型/小模型切换）。

4. **能力层**  
Prompt 模板中心、Tool 调用、RAG 检索服务（向量+关键词混检）。

5. **数据层**  
MySQL（用户/订单/任务状态）、Redis（会话和缓存）、对象存储（文件/产物）、向量库（知识检索）。  
关键写操作必须幂等键，避免重试导致重复副作用。

6. **安全与治理层**  
输入输出内容安全（涉政/涉黄/隐私）、提示词注入防护、工具白名单、最小权限、审计日志全链路可追溯。

7. **可观测与运维层**  
Metrics + Logs + Traces 三件套；重点盯成功率、时延、成本、幻觉率、工具失败率。  
上线策略：可灰度、可监控、可回滚。
