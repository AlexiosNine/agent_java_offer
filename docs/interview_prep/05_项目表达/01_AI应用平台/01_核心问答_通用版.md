# AI应用平台核心问答（通用版）

## 使用说明

- 本文件整理了 AI 应用平台（如 AI 编程助手、AI 代码生成平台、企业 AI 应用平台）面试中的高频问题
- 内容已通用化，适用于任何 AI 应用平台方向的面试准备
- 问题按技术主题组织，而非按特定公司

## 核心技术主题

### 1. Agent 架构设计

**问题：如何设计一个生产级的 Agent 系统架构？**

**口述答案（300-400字）**：

我会按"控制面 + 数据面 + 治理面"三层来设计。

控制面负责任务编排：Planner 做任务分解，Executor 执行具体步骤，Tool Router 路由工具调用，Evaluator 评估结果质量，Retry/Fallback 处理异常。这几层要解耦，避免单点故障。

数据面负责知识和记忆：Memory 模块管理短期和长期记忆，Observation 模块收集环境反馈，RAG 模块提供外部知识检索。关键是做好上下文管理，避免信息过载。

治理面负责可靠性：权限控制（RBAC/ABAC）、审计日志（全链路 trace）、监控告警（成功率、延迟、成本）、灰度发布和降级开关。

**Single Agent vs Multi-Agent 选型**：
- 单 Agent：任务简单、流程固定、上下文可控
- 多 Agent：任务复杂、需要专业化分工、需要并发执行

多 Agent 的核心收益不是"更酷"，而是：
1. **上下文隔离**：每个子 Agent 只看必要信息，减少幻觉
2. **职责专业化**：检索、执行、审校、风控各司其职
3. **故障隔离**：局部失败不拖垮全局
4. **权限最小化**：不同 Agent 不同权限，安全边界清晰
5. **成本优化**：简单任务用小模型，关键步骤用强模型

**常见追问**：
- 如何处理 Agent 之间的通信和协作？
- 如何避免 Agent 陷入死循环？
- 如何做 Agent 的可观测性？

### 2. RAG 与知识库设计

**问题：如何设计一个生产级的 RAG 系统？**

**口述答案（300-400字）**：

完整链路分离线和在线两段。

**离线阶段**：
1. 文档清洗：去噪声、去重、格式统一
2. 切块策略：按语义边界切分，保留 10%-20% 重叠
3. Embedding：选择合适的模型（中文 BGE/M3E，英文 OpenAI Ada）
4. 索引构建：向量索引 + 元数据索引，支持混合检索

**在线阶段**：
1. 查询改写：补全指代、规范术语、拆分复合问题
2. 混合检索：向量检索（语义） + 关键词检索（精确匹配）
3. 重排序：用 Cross-Encoder 对 Top-50 候选精细打分，选出 Top-5
4. 上下文组装：拼接检索结果 + 用户问题 + 输出约束
5. 生成回答：调用 LLM，要求带引用来源

**规则知识库 vs Embedding Memory**：
- 规则库：硬约束，适合风控、权限、计费等需要精确匹配的场景
- Embedding：软记忆，适合 FAQ、经验复用、知识问答等需要语义理解的场景
- 工程实践：规则库做"最终裁决"，Embedding 做"补充上下文"

**知识过期/错误治理**：
1. 入库时：版本号、生效/失效时间、来源等级、责任人
2. 检索时：元数据过滤 + 新鲜度打分 + 可信度打分
3. 生成时：要求带引用，证据不足时拒答
4. 线上时：快速封禁问题文档，监控过期命中率

**常见追问**：
- 如何选择切块大小和重叠长度？
- 如何评估检索质量（Recall@K, nDCG）？
- 向量数据库如何选型（FAISS vs Milvus vs pgvector）？

### 3. 大代码库理解

**问题：如何让 AI 快速理解几十万行代码的项目？**

**口述答案（300-400字）**：

我不会让 AI 一次性读几十万行代码，而是采用"渐进式披露"策略。

**第一步：构建项目地图**
- 目录结构图谱：入口、核心业务、数据层、外部接口
- 模块摘要：每个模块的职责和关键类
- 关键调用链：主流程的执行路径

**第二步：按需检索**
- 根据任务只给相关的 5-10 个文件
- 使用语义检索找到相关代码片段
- 避免全仓库上下文污染

**第三步：增量索引**
- 对代码库做向量化索引
- 支持语义搜索和符号搜索
- 变更时增量更新索引

**第四步：验证与沉淀**
- AI 的理解需要通过调用关系验证
- 确认后的结论沉淀成文档（模块说明、关键入口、常见坑）
- 下次查询直接复用

**核心原则**：先搭框架，再看细节；渐进式披露；每步验证；沉淀摘要。

**常见追问**：
- 如何处理代码变更的影响分析？
- 如何识别核心 domain object？
- 如何处理跨语言项目？

### 4. AI 代码生成质量保障

**问题：AI 生成的代码"看起来对但逻辑错"，如何发现和兜底？**

**口述答案（300-400字）**：

**根因分析**：
1. 上下文缺失：没有看到相关代码和约束
2. 需求不清：PRD 模糊或有歧义
3. 约束没喂：没有提供代码规范和安全规则
4. 检索不准：相关代码没有被检索到
5. 反馈不够：工具执行结果没有回传
6. 评测缺失：没有自动化测试验证

**治理策略**：

**写代码前**：
- 引入 Spec 规范约束（不能做什么）
- 提供正确范例和代码规约
- 任务切小，降低复杂度

**写代码时**：
- 先做影响分析，识别相关文件
- 规则库 + 检索增强上下文
- 计划先行，不要直接出码

**写代码后**：
- 双 Agent review（一个写，一个审）
- 单测 + 集测自动化验证
- 边界场景和异常场景覆盖

**上线前**：
- 小流量灰度，看真实业务指标
- Feature Flag 控制，异常可一键关闭
- 写操作加幂等键，避免重复副作用
- 审计日志 + request_id，可回放定位

**常见追问**：
- 如何提高代码留用率？
- 如何处理复杂业务逻辑？
- 如何做代码质量评测？

### 5. MCP 与工具调用

**问题：如何设计 MCP / Tool Calling 的权限控制、重试和标准化？**

**口述答案（300-400字）**：

我们不让模型直接"自由调用"工具，而是经过一个 **Tool Gateway**。

**第一层：权限控制**
- 按用户、Agent、场景做 RBAC/ABAC
- 默认最小权限，只开放白名单工具
- 高危工具（下单、转账、删数据）必须二次确认
- 参数做 schema 校验和额度限制

**第二层：失败重试**
- 错误分类：只有超时、限流、临时 5xx 才重试
- 业务错误、参数错误、权限错误不重试
- 重试策略：指数退避 + 抖动
- 设置最大重试次数和总超时
- 配置熔断与降级，防止雪崩
- 写操作必须幂等键

**第三层：结果标准化**
- 统一 envelope：`{code, message, data, retryable, trace_id}`
- 外部错误码映射成平台标准错误码
- 模型只消费标准字段，不依赖第三方原始响应

**价值**：
- 权限可控：安全边界清晰
- 重试可预期：不会造成副作用
- 结果可编排：统一接口便于流程控制
- 可观测：全链路 trace，便于监控和审计

**核心原则**：模型只负责决策，工具网关负责安全与可靠性。

**常见追问**：
- 如何处理工具调用超时？
- 如何做工具的版本管理？
- 如何处理工具依赖关系？

### 6. 评测与质量监控

**问题：如何建立 AI 应用的评测和监控体系？**

**口述答案（300-400字）**：

按"**离线评测 + 线上监控 + 回归闭环**"三段式设计。

**离线评测（分层）**：
1. 任务成功率：是否完成目标
2. 过程质量：工具调用是否正确、是否越权、是否死循环
3. 安全鲁棒性：注入攻击、脏数据、长尾问题

数据集要覆盖：
- 真实用户样本（主流场景）
- 边界样本（极端情况）
- 故障样本（异常处理）
- 黄金集（版本对比基准）

**线上监控（四类指标）**：
1. 效果指标：成功率、人工接管率、用户满意度
2. 效率指标：P50/P95 延迟、步骤数、token 成本
3. 稳定性指标：工具失败率、重试率、超时率、回环率
4. 安全指标：越权调用、拒答命中、敏感操作拦截

**可观测性**：
- 全链路 trace_id：串联"计划-调用-结果-答复"
- 实时告警：指标异常立即通知
- 一键降级：异常时先止损再排查

**回归闭环**：
- 线上 bad case 回流到离线评测集
- 持续 A/B 测试和回归测试
- 迭代优化 prompt、检索、工具

**常见追问**：
- 如何评估 RAG 的检索质量？
- 如何做 Agent 的可解释性？
- 如何处理评测数据的标注成本？

### 7. 灰度发布与在线双跑

**问题：如何设计灰度切流、在线双跑和回滚机制？**

**口述答案（300-400字）**：

**灰度切流**：
1. 流量分层：按用户 ID、地域、设备等维度分流
2. 小流量起步：1% → 5% → 10% → 50% → 100%
3. 观察指标：成功率、延迟、错误率、用户反馈
4. 自动/手动切流：指标正常自动放量，异常自动回滚

**在线双跑**：
1. 新旧链路并行：同一请求同时走新旧两条链路
2. 只返回旧链路结果：不影响真实用户
3. 结果比对：记录新旧结果差异
4. 日志和埋点：全量记录用于分析

**回滚机制**：
1. Feature Flag：一键关闭新功能
2. 版本管理：保留旧版本代码和配置
3. 数据回滚：数据库变更支持回滚
4. 快速止损：异常时 5 分钟内完成回滚

**关键设计**：
- 新旧链路隔离：避免相互影响
- 幂等性保证：双跑不会造成副作用
- 可观测性：全链路 trace，便于定位问题
- 自动化：减少人工介入，提高响应速度

**常见追问**：
- 如何处理数据库 schema 变更？
- 如何做流量录制和回放？
- 如何评估灰度效果？

### 8. 基础技术能力

**必备知识点**：

1. **Transformer 架构**：Self-Attention、Multi-Head Attention、Position Encoding
2. **向量检索算法**：HNSW、IVF、PQ、ANN 原理
3. **缓存一致性**：Cache-Aside、Write-Through、Write-Behind
4. **分布式系统**：CAP、BASE、最终一致性
5. **网络基础**：HTTP/HTTPS、TCP/IP、WebSocket
6. **Linux/Git**：常用命令、分支管理、冲突解决
7. **算法题**：中等难度，数组、链表、树、动态规划

**面试建议**：
- 不要只会背概念，要能讲清楚原理和应用场景
- 结合实际项目经验，说明如何解决具体问题
- 准备 1-2 个深入的技术点，能展开讲 10 分钟

## 面试准备建议

### 优先级排序

**P0（必须准备）**：
1. Agent 架构设计
2. RAG 与知识库
3. 代码生成质量保障
4. 评测与监控

**P1（重点准备）**：
5. MCP 与工具调用
6. 大代码库理解
7. 灰度发布与双跑

**P2（了解即可）**：
8. 基础技术能力

### 准备策略

1. **先理解原理**：不要死记硬背，理解为什么这么设计
2. **结合项目经验**：每个问题都准备 1-2 个实际案例
3. **准备追问**：每个主问题准备 3-5 个常见追问
4. **模拟面试**：找朋友或录音自己讲一遍，找出卡壳的地方
5. **持续更新**：AI 领域变化快，定期更新知识

### 口述技巧

1. **结构化表达**：先讲框架，再讲细节
2. **数据支撑**：用具体数字说明效果（成功率、延迟、成本）
3. **权衡分析**：说明不同方案的优劣势和选型依据
4. **问题导向**：说明解决了什么问题，而不是只讲技术
5. **可落地性**：说明如何在生产环境实施，而不是只讲理论
