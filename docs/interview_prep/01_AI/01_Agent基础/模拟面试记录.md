## 第一题：什么是基于 LLM 的 Agent

**面试官提问**：
> 请你用 2-3 分钟时间，解释一下什么是基于 LLM 的 Agent，它和普通的大模型对话有什么本质区别？

**考生回答**：
> 基于 LLM 的 Agent 一般指的是使用大语言模型作为决策中心的智能体，相比于普通的大模型，本质区别在于 agent 是面向具体的任务，在 LLM 的智能外，必须具备执行（Action）、观察（observation）的能力。可以说 Agent = LLM + 工具执行。

**综合评分**：6/10

### A：知识点确认（/5）
- **得分**：2.5/5
- **掌握**：Agent = LLM + 工具执行；Action + Observation 概念
- **遗漏**：四层架构（规划/记忆/工具/执行控制）；与普通对话的多维对比；Planning 和 Memory 组件

### B：结构化表达（/5）
- **得分**：3.5/5
- **亮点**：回答简洁，没有冗余
- **问题**：无分层结构（是什么→为什么→怎么做）；缺具体场景举例；"面向具体任务"表达模糊

---

**优点**：
- ✅ 抓住了核心差异：Agent = LLM + 工具执行
- ✅ 提到了 Action 和 Observation，体现了对 ReAct 框架的理解
- ✅ 回答简洁，没有冗余

**主要问题**：

1. **框架不完整（-2分）**
   - 只提到了"工具执行"和"观察"，缺少了 Agent 的其他核心组件
   - 没有提到规划（Planning）、记忆（Memory）、状态管理
   - 给人感觉 Agent 只是"LLM + 工具调用"，这个理解太浅

2. **缺少对比维度（-1分）**
   - 没有说清楚"普通大模型对话"是什么样的
   - 对比不够立体，只说了"能执行"，没说"为什么需要执行"

3. **表达不够结构化（-1分）**
   - 没有分层次讲（是什么 → 为什么 → 怎么做）
   - "面向具体的任务"这句话太模糊，普通对话也可以面向任务

**优化建议**：

建议的回答框架：
```
1. 定义（30秒）
   - Agent 是什么：以 LLM 为决策中枢的闭环执行系统
   - 核心组件：规划、记忆、工具、执行控制

2. 对比（30秒）
   - 普通对话：单轮问答，无状态，无执行能力
   - Agent：多轮闭环，有状态，能感知-决策-行动-反馈

3. 举例（30秒）
   - 普通对话："帮我查一下北京天气" → 只能回答"我不知道"
   - Agent："帮我查一下北京天气" → 调用天气 API → 返回实时结果

4. 关键差异（30秒）
   - 从"回答问题"到"完成任务"
   - 从"封闭知识"到"开放能力"
```

**改进版参考答案**：
> 我会把 Agent 定义为"以 LLM 为决策中枢、可在环境中闭环执行任务的系统"。它通常由四层构成：规划层负责任务分解，记忆层存储上下文和知识，工具层调用外部能力，执行控制层管理状态和异常。
>
> 和普通对话模型相比，关键差异是 Agent 能持续感知、决策、行动并根据反馈修正。比如问"北京天气"，普通模型只能说"我不知道"，而 Agent 会调用天气 API 拿到实时数据再回答。本质上是从"回答问题"升级到"完成任务"，从封闭知识升级到开放能力。

---

## 复盘要点

1. **记住四层架构**：规划、记忆、工具、执行控制
2. **用对比说清楚差异**：不只是"能做什么"，还要说"为什么需要"
3. **举具体例子**：抽象概念要用场景落地
4. **结构化表达**：定义 → 对比 → 举例 → 总结

## 第二题：ReAct 框架

**面试官提问**：
> ReAct 框架是什么？它是如何将推理和行动结合起来的？请结合一个具体场景说明 ReAct 的执行流程。

**考生回答**：
> ReAct 框架指的是智能体的一种组成架构，一般来说 ReAct 指智能体执行的步骤由规划（Reasoning）+ 执行（Action）两阶段构成的，智能体通过不断循环 ReAct 流程最终闭环完成任务。每一轮的 ReAct 执行完之后，智能体通过观察上下文/环境变化，作为新的依据来进行下一轮的规划+执行。
>
> 例如智能体在 Coding 中，Agent 先通过观察当前的代码仓库了解项目背景和进度，再规划如何实现新功能，在初步规划后，执行 write/edit 操作更新代码；下轮再重复这一过程，直至模型认为新功能已经完全实现。

**综合评分**：7.5/10

### A：知识点确认（/5）
- **得分**：3.5/5
- **掌握**：Reasoning + Action 循环；闭环思维；具体场景（Coding Agent）
- **遗漏**：Observation 作为独立第三步；与 CoT 的区别；"推理"vs"规划"的概念偏差

### B：结构化表达（/5）
- **得分**：4/5
- **亮点**：有具体场景；描述了完整循环流程
- **问题**：场景描述太笼统，没有展示每轮输入输出；缺与其他框架的对比

---

**优点**：
- ✅ 正确理解了 ReAct 的核心：Reasoning + Action 的循环
- ✅ 提到了"观察上下文/环境变化"作为下一轮输入，体现了闭环思维
- ✅ 给出了具体场景（Coding Agent），有落地感
- ✅ 描述了完整的循环流程

**主要问题**：

1. **概念理解有偏差（-1分）**
   - ReAct 不是"规划 + 执行"两阶段，而是 **Reasoning（推理）+ Action（行动）+ Observation（观察）** 三步循环
   - 把 Reasoning 翻译成了"规划"，但 ReAct 论文中的 Reasoning 更强调"思考过程"而非"任务分解"
   - 缺少了 **Observation** 这个关键步骤的显式说明

2. **场景举例不够具体（-1分）**
   - Coding 场景的描述太笼统："观察代码仓库 → 规划 → 写代码"
   - 没有展示每一步的具体输入输出，看不出 ReAct 的"交替推理"特点
   - 缺少"中间失败后如何调整"的例子

3. **缺少与其他框架的对比（-0.5分）**
   - 没有说明 ReAct 和传统 Chain-of-Thought 的区别
   - 没有说明为什么需要"推理和行动交替"而不是"先推理完再行动"

**优化建议**：

建议的回答框架：
```
1. 定义（30秒）
   - ReAct = Reasoning + Action + Observation 的循环
   - 核心思想：推理和行动交替进行，而不是分离

2. 与 CoT 对比（20秒）
   - CoT：先推理完再给答案（封闭推理）
   - ReAct：边推理边行动，根据观察调整（开放推理）

3. 执行流程（40秒）
   - Thought：分析当前状态，决定下一步
   - Action：执行工具调用
   - Observation：获取执行结果
   - 循环直到任务完成

4. 具体场景（60秒）
   - 用一个完整的例子展示 3-5 轮循环
   - 包含中间失败和调整的过程
```

**改进版参考答案**：
> ReAct 是 Reasoning + Action + Observation 的缩写，核心思想是让 LLM 的推理和行动交替进行，而不是分离。和传统的 Chain-of-Thought 不同，CoT 是封闭推理（先想完再答），ReAct 是开放推理（边想边做，根据反馈调整）。
>
> 执行流程是一个循环：每轮先 Thought（分析当前状态，决定下一步），再 Action（调用工具），然后 Observation（获取执行结果），把结果加入上下文后进入下一轮，直到任务完成。
>
> 举个例子，用户问"北京明天会下雨吗？如果下雨帮我改约会时间"：
> - **轮1** - Thought: 需要先查天气 → Action: call_weather_api("北京", "明天") → Obs: 明天有雨
> - **轮2** - Thought: 确实下雨，需要查日历 → Action: get_calendar() → Obs: 明天下午3点有约会
> - **轮3** - Thought: 需要改到后天 → Action: reschedule("约会", "后天3点") → Obs: 改约成功
> - **轮4** - Thought: 任务完成 → Action: Finish → 返回结果
>
> 关键是每轮的 Observation 会影响下一轮的 Thought，这样 Agent 可以根据实际情况动态调整策略，而不是一开始就把所有步骤规划死。

**复盘要点**：
1. **记住三步循环**：Thought → Action → Observation（不是两步）
2. **对比 CoT**：说清楚"交替推理"和"封闭推理"的区别
3. **举例要具体**：展示 3-5 轮完整循环，包含中间调整
4. **强调动态性**：根据观察调整策略，而不是预先规划

## 第三题：工具调用超时处理

**面试官提问**：
> 在生产环境中，Agent 的 Tool Calling 经常会遇到各种异常情况。假设你负责设计一个 Agent 系统，某个工具调用超时了，你会怎么处理？请从工程角度详细说明你的异常处理策略。

**考生回答**：
> 工具调用超时的处理机制一般来说我们会针对工具的性质，所耗资源来做多层次的超时处理策略。
>
> 对于短执行时间/轻量任务，我们会采取重试策略，借助随机抖动/指数退避重试来打散请求，避免下游请求量突增。
>
> 对于长执行时间/重量任务，我们会采取异步执行，接受回调的方式，来进行重试，并且有对应的降级策略，包括使用能达到近似效果的轻量级工具，例如某些复杂的分析工具降级使用 websearch 工具，实时信息改为调用缓存等等。
>
> 对于非关键链路上的工具超时，除了上述策略外，我们还可以使用直接返回默认信息，或者说直接返回超时报错，告知模型，LLM 可以通过上下文或者其他工具的返回补全信息。
>
> 如果是关键路径上的工具超时，在我们上述的一些策略都失败，且有写操作的情况下，则让模型直接向用户返回失败信息，避免用户的误解，但是这样要求我们的写操作的工具做好幂等或者数据一致性的处理；例如类似订票，付款这些关键性的写操作。

**综合评分**：8/10

### A：知识点确认（/5）
- **得分**：4/5
- **掌握**：指数退避+随机抖动；幂等性；异步回调；降级策略；分层分类思维
- **遗漏**：熔断器（Circuit Breaker）机制；可观测性三件套（Metrics/Trace/Log）

### B：结构化表达（/5）
- **得分**：4/5
- **亮点**：按任务类型分类处理；有具体例子（websearch降级、订票付款）
- **问题**：策略是清单式平铺，缺决策树（"第一步判断X，如果Y走A"）

---

**优点**：
- ✅ **分层思维清晰**：按任务类型（轻/重、关键/非关键）分类处理，体现了工程判断力
- ✅ **技术细节到位**：提到了指数退避、随机抖动、异步回调、降级策略
- ✅ **考虑了写操作的特殊性**：强调幂等性和数据一致性，说明有生产经验
- ✅ **给出了具体例子**：websearch 降级、缓存替代、订票付款场景

**主要问题**：

1. **缺少完整的决策树（-1分）**
   - 回答是"平铺"的策略列举，没有形成清晰的决策流程
   - 面试官问"你会怎么处理"，期望的是一个 if-else 决策树，而不是策略清单
   - 建议用"第一步判断 X，如果 Y 则走策略 A，否则走策略 B"的结构

2. **缺少监控和可观测性（-0.5分）**
   - 只讲了"怎么处理超时"，没讲"怎么发现超时是系统性问题还是偶发问题"
   - 生产环境需要 metrics（超时率、重试率）和告警，这是工程落地的关键
   - 没有提到如何定位是哪个环节超时（网络、下游服务、参数问题）

3. **降级策略不够具体（-0.5分）**
   - "复杂分析工具降级用 websearch"这个例子不太合理（websearch 也可能超时）
   - 降级应该是"功能降级"而不是"工具替换"，比如：
     - 实时数据 → 缓存数据（时效性降级）
     - 精确结果 → 近似结果（准确性降级）
     - 完整信息 → 部分信息（完整性降级）

**优化建议**：

建议的回答框架：
```
1. 超时判定与分类（20秒）
   - 静态超时 vs 动态超时
   - 根据工具类型分类：读/写、轻/重、关键/非关键

2. 分层处理策略（60秒）
   - 第一层：重试（指数退避 + 抖动）
   - 第二层：降级（功能降级、备选工具）
   - 第三层：熔断（避免雪崩）
   - 第四层：转人工（关键操作失败）

3. 特殊场景处理（30秒）
   - 写操作：幂等性保证
   - 异步任务：回调机制
   - 并行调用：部分失败处理

4. 可观测性（20秒）
   - 监控指标：超时率、重试率、降级率
   - 告警策略：连续失败 N 次触发
   - 定位手段：trace_id 串联全链路
```

**改进版参考答案**：
> 我会按"判定 → 分层处理 → 监控"的思路来设计。
>
> **第一步，超时判定与分类**：
> - 超时阈值分静态（固定 5 秒）和动态（根据历史 P95 设置）
> - 工具分类：读操作（查询、搜索）vs 写操作（创建、修改）、轻量（<1秒）vs 重量（>5秒）、关键路径 vs 非关键路径
>
> **第二步，分层处理策略**：
> - **第一层：重试**。适用于轻量读操作和临时性故障。策略是指数退避 + 随机抖动（第 1 次等 1 秒，第 2 次等 2-4 秒，第 3 次等 4-8 秒），最多重试 3 次，总超时 30 秒。
> - **第二层：降级**。重试失败后走降级：实时数据降级用缓存（时效性降级）、精确查询降级用模糊匹配（准确性降级）、完整结果降级返回 top-K（完整性降级）。
> - **第三层：熔断**。某个工具连续失败 10 次，打开熔断器，后续请求直接拒绝不调用，60 秒后进入半开状态尝试恢复。避免慢工具拖垮整个系统。
> - **第四层：转人工**。关键路径的写操作（如订票、付款）降级失败后，直接返回失败信息给用户，不要让 Agent 自作主张。前提是写操作做好幂等性（带幂等键）和事务性（要么全成功要么全失败）。
>
> **第三步，特殊场景**：
> - 重量任务（如大数据分析）用异步执行 + 回调，不阻塞主流程
> - 并行调用多个工具时，允许部分失败，只要核心工具成功就继续
>
> **第四步，可观测性**：
> - 监控指标：工具超时率、重试率、降级率、熔断触发次数
> - 告警策略：某工具超时率 > 10% 或连续失败 5 次触发告警
> - 定位手段：每次调用打 trace_id，串联 LLM 推理 → 工具调用 → 下游服务，快速定位是网络问题、参数问题还是下游服务问题

**复盘要点**：
1. **用决策树而非清单**：第一步判断 X，如果 Y 则走策略 A
2. **四层防御体系**：重试 → 降级 → 熔断 → 转人工
3. **降级是功能降级**：时效性/准确性/完整性降级，而不是工具替换
4. **可观测性是关键**：监控 + 告警 + trace，快速定位问题

## 第四题：Agent 产品设计

**面试官提问**：
> 如果让你从零开始设计一个 Agent 产品，你会怎么切入？选什么场景？怎么定义成功指标？怎么分阶段落地？

**考生回答**：
> 从零开始设计 Agent 产品，我会从高频率、高需求以及重复性这三点来进行分析，并选择具体的场景来进行切入。
>
> 选取场景的话，会从一些工作场景下出发，具体可以是会议助手、文案助手以及企业客服等场景。这几个场景都是工作中高频率出现，实际需求旺盛并且工作流程比较清晰，重复性工作较多的场景。
>
> 以企业客服为例，成功指标可以定义成：用户满意度、人工介入率、任务成功率等指标。
>
> 用户满意度用于衡量用户对于企业服务回复的满意程度，评估可以从人工反馈、LLM 异步评估出发。
>
> 人工介入率用于衡量企业客服的任务闭环能力，衡量智能体能够在多大程度上替代传统的人工客服，定义智能体能力的边界以及后续优化的重要指标。
>
> 任务成功率用于衡量企业客服智能体的处理能力，从稳定性和可控性两方面出发，表现了智能体以及对应工具/下游服务的稳定性，智能体处理用户问题的可控性，避免智能体发散。

**考生回答**：

> 记忆压缩时机：
> 1. 上下文达到一定程度，触发自动压缩，LLM 评估重要信息保存至长期记忆。
> 2. 任务执行完成/定时任务，异步执行记忆压缩和转移，将重要信息保存到长期记忆中。

> 1. 规则匹配，例如角色扮演中的人设，用户的住址/性别，生日这些重要信息，根据规则匹配保留。

> 我会设计三层记忆：短期记忆用 Redis 存当前会话的最近 20 轮对话，长期记忆用向量库 + 关系型数据库存用户偏好和历史决策，工作记忆存任务执行状态。

## 第六题：评估体系设计

**面试官提问**：
> 如果你做了一个 Agent 系统，你会怎么设计它的评估体系？请分别说说：离线评估、在线评估、以及如何定位问题到底出在 Prompt、RAG、还是工具调用。

**考生回答**：
> 评估体系的设计我们要自顶向下，从我们的实际业务场景以及要解决的问题出发，结合 agent 领域的一些指标制定出一套可量化的评估体系。
>
> 例如在智能客服场景中，我们的评估体系有以下几种：
> 1. 北极星体系：用户满意度，人工介入率，用于衡量整个智能客服系统的任务闭环程度和用户反馈。
> 2. 技术指标：调用量，任务成功率，任务执行消耗轮次，执行时间。用于衡量智能体在该场景下的稳定性和可控性。
> 3. 成本指标：消耗 token 量级，人力节省成本，用于衡量智能体的投入产出比。
>
> 离线评估，一般是通过数仓链路，离线捞取 agent 的日志，通过日志重放让 LLM 来对整个系统的各个指标进行评估。一般用于请求量高，耗时时间长，不需要在线回溯的场景。
>
> 在线评估，一般是在 agent 执行任务的过程中，由人工或者 agent 来实时评估相应的各个指标，往往需要结合多 Agent 协同, Reflection 机制。一般用于请求量较低，耗时时间较短，实时性要求较低，高敏感权限的场景。
>
> 除此以外，评估系统必须结合 agent 可观测性，来帮助我们定位瓶颈，例如 trace 以及 agent 的 trajectory log，追溯智能体每一轮的推理和执行过程，来发现问题出现的环节。

### A：知识点确认（/5）

- **掌握**：指标体系分层（北极星/技术/成本）；trace 和 trajectory log 概念
- **遗漏**：控制变量法（核心！）；golden test set；A/B 测试；如何隔离 Prompt/RAG/工具问题

### B：结构化表达（/5）
- **得分**：3.5/5
- **亮点**：指标体系清晰；场景化思维（智能客服）；离线/在线有区分
- **问题**：核心问题（问题定位方法）只用一句话带过，占比严重失衡；缺决策树

**优点**：
- ✅ 指标体系分层清晰：北极星指标 → 技术指标 → 成本指标
- ✅ 场景化思维：以智能客服为例，指标选择贴合实际业务
- ✅ 提到了可观测性：trace 和 trajectory log 方向正确

   - 应该用**控制变量法**隔离问题：固定两个变量，改变一个，看哪个导致问题

2. **离线/在线评估方法过于抽象（-1分）**
   - 离线评估：未提及 golden test set、回归测试、LLM-as-judge
   - 在线评估：未提及 A/B 测试、采样标注、灰度发布

**改进版参考答案**：
> **一、离线评估**：构建 golden test set（标注数据集），包含典型场景和边界 case。每次迭代跑回归测试，用 LLM-as-judge 批量评估输出质量，监控任务成功率、检索召回率、工具调用准确率。
>
> **二、在线评估**：通过 A/B 测试对比新旧版本，实时采样部分请求做人工标注，结合灰度发布逐步放量。关键指标包括用户满意度、人工介入率、任务完成时长。

> - **Prompt 问题**：同样的检索结果和工具，换 Prompt 后输出变好 → Prompt 问题

> - **工具调用问题**：用 mock 工具结果替换，输出变好 → 工具问题，查工具成功率
>

**复盘要点**：
1. **控制变量法是核心**：固定两个变量改一个，定位是 Prompt / RAG / 工具的问题
2. **离线评估要有 golden test set**：不只是"日志重放"，要有标注数据集和回归测试
3. **在线评估要有 A/B 测试**：不只是"人工评估"，要有对比实验和灰度发布
4. **题目问什么就重点答什么**：问题定位方法是这道题的核心，应该占 50% 以上的篇幅
