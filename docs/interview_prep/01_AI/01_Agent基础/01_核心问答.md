## 1. 主干问题：什么是基于 LLM 的 Agent，它和普通大模型有什么区别？

### 子问题：Agent 定义与核心组件

**主问题**：你如何定义基于 LLM 的 Agent？核心组件有哪些？

**口述答案**：
我会把 Agent 定义为"以 LLM 为决策中枢、可在环境中闭环执行任务的系统"。它通常由四层构成：

1. **规划层（Planning）**：任务分解、步骤规划、目标设定
2. **记忆层（Memory）**：短期记忆（对话历史）、长期记忆（知识库）、工作记忆（任务状态）
3. **工具层（Tools）**：外部能力调用（搜索、数据库、API）
4. **执行控制层（Control）**：状态管理、异常恢复、可观测性

和普通对话模型相比，关键差异是它能持续感知、决策、行动并根据反馈修正。若从软件 Agent 走向具身或仿真 Agent，难点会从"文本理解"升级到"高噪声感知 + 连续动作控制 + 安全约束"，系统设计重点也会从回答质量转为"安全、稳定、可恢复"。

### 子问题：LLM 和 Agent 有什么区别？

**口述答案**：
我会从三个层面区分 LLM 和 Agent。

第一，定位不同。LLM 本质上是一个模型，它擅长理解语言、推理和生成内容。用户给它一个 prompt，它根据上下文生成回答。而 Agent 更像是一个系统，它以 LLM 为核心，但不只是生成文本，而是围绕一个目标去规划步骤、调用工具、执行动作，并根据结果继续调整。

第二，能力边界不同。普通 LLM 通常只能基于已有上下文回答问题。比如你问它怎么排查接口延迟升高，它可以给你排查思路。但 Agent 可以连接外部工具，比如监控系统、日志平台、数据库、代码仓库。它可以真的去查指标、看日志、对比发布记录，然后汇总原因。

第三，执行方式不同。LLM 更偏向一次性生成答案；Agent 通常是一个循环过程：先理解目标，再拆解任务，然后执行一步，观察结果，再决定下一步。也就是类似于 plan、act、observe、adjust 这样的闭环。

从工程角度看，我会把 Agent 理解成：LLM 加上工具调用、任务规划、状态管理和安全控制之后形成的任务执行系统。

当然，落地 Agent 的时候不能只关注它能不能调用工具，还要考虑可靠性，比如工具参数校验、权限控制、失败重试、最大执行轮数、人工确认和审计日志。因为 Agent 一旦能执行真实操作，风险就比普通 LLM 高很多。

一句话总结：LLM 主要解决"如何理解和生成"，而 Agent 解决的是"如何基于目标完成一系列动作"。LLM 是 Agent 的核心能力，但 Agent 是一个更完整的工程系统。

**常见追问**：
1. Agent 与 Workflow 最大差别是什么？
2. 为什么说"可观测性"比"会答题"更重要？
3. 具身场景为什么更强调实时性与安全兜底？

### 子问题：Agent 和 Workflow 的区别是什么？

**口述答案**：
我会把 workflow 理解成一个预先定义好的流程系统，把 Agent 理解成一个可以根据目标和环境反馈动态决策的执行系统。更直白地说：Workflow 是人提前设计好步骤，系统按流程执行；Agent 是给它一个目标，它自己决定下一步该做什么。

比如一个报销审批流程：先提交申请 → 直属经理审批 → 财务审核 → 打款。这就是典型 workflow，步骤、分支、规则基本都是提前定义好的，系统主要负责按流程推进。但如果是一个 Agent，比如让它"帮我分析这个季度销售下滑的原因"，它可能会自己决定先查销售数据，再看地区维度，再看产品维度，再查市场变化，最后生成报告。这个过程不是完全预先写死的，而是根据中间结果动态调整。

我会从三个角度区分：

第一，确定性不同。Workflow 通常是确定性的，流程节点、条件分支、失败处理都是提前设计好的。Agent 更偏动态决策，它会根据当前上下文、工具返回结果和目标，决定下一步动作。

第二，控制权不同。Workflow 的控制权主要在人，工程师提前定义好流程。Agent 的一部分控制权交给了模型，让模型做任务拆解、工具选择和下一步决策。

第三，适用场景不同。Workflow 适合规则清晰、流程稳定、对可靠性要求高的场景，比如审批流、订单处理、CI/CD pipeline。Agent 适合目标明确但路径不确定的场景，比如问题排查、研究分析、复杂客服、多步骤代码修复。

一句话总结：Workflow 是 predefined steps，Agent 是 dynamic decision-making。

但工程落地里，我不会认为两者是对立的。很多实际系统会把它们结合起来：外层用 workflow 控制大流程，保证稳定性和可观测性；在某些不确定节点里嵌入 Agent，让它处理需要推理和探索的部分。

比如线上故障排查系统可以这样设计：
1. Workflow 先固定执行基础检查，比如查报警、查部署、查核心指标；
2. 如果基础检查无法定位问题，再让 Agent 根据日志、trace、指标动态探索；
3. 最后回到 workflow，要求人工确认后才能执行修复动作。

所以我的最终理解是：Workflow 更强调流程可控和稳定执行；Agent 更强调自主规划和动态适应。生产环境里，最靠谱的方式通常不是纯 Agent，而是 workflow 加 Agent 的混合架构。

## 2. 主干问题：ReAct 是什么，Agent 的规划能力怎么设计？

### 子问题：ReAct 框架详解

**主问题**：请详细解释 ReAct 框架。它是如何将思维链和行动结合起来的？

**口述答案**：
ReAct 的本质是把"推理"和"行动"交替执行，形成 **Thought-Action-Observation** 的循环：

1. **Thought（思考）**：模型分析当前状态，决定下一步做什么
2. **Action（行动）**：调用工具执行具体动作
3. **Observation（观察）**：获取工具返回结果，更新状态
4. 循环直到任务完成

它相比静态 CoT 的优势是可交互、可纠错、可解释，适合信息不完整或环境会变化的任务；代价是链路更长，线上要控制步数、超时和成本。

**规划能力对比**：
- **CoT（Chain of Thought）**：单路径分步推理，成本低但容错弱
- **ToT（Tree of Thoughts）**：树状分支探索和回溯，成功率高但算力开销大
- **GoT（Graph of Thoughts）**：图结构允许分支合并和循环优化，适合复杂依赖

**工程实践**：建议把"规划"和"执行"解耦：Planner 负责拆解，Executor 负责落地，减少模型一步到位失败。

**常见追问**：
1. 什么时候不该用复杂规划而该用固定流程？
2. 规划深度如何控制成本？
3. 如何避免"想太多、做太少"？

## 3. 主干问题：Tool Calling 机制如何设计？

### 子问题：Tool Calling 完整流程

**主问题**：如何设计一个生产级的 Tool Calling 系统？

**口述答案（300-400字）**：

Tool Calling 的核心是"模型决策 + 工具网关执行"，分四个阶段：

**1. 函数定义与注册**
- 用 JSON Schema 定义工具签名（名称、描述、参数、返回值）
- 注册到工具注册表，支持版本管理和动态加载
- 示例：
```json
{
  "name": "search",
  "description": "搜索相关信息",
  "parameters": {
    "query": {"type": "string", "description": "搜索关键词"}
  }
}
```

**2. 模型调用与参数解析**
- 模型根据任务决定调用哪个工具
- 生成结构化的函数调用请求（function name + arguments）
- 解析模型输出，提取工具名和参数（JSON 解析或正则匹配）

**3. 工具执行与结果处理**
- 通过工具网关执行（不是模型直接调用）
- 参数校验：类型检查、范围检查、权限检查
- 执行工具并捕获异常
- 标准化返回格式：`{code, message, data, retryable, trace_id}`

**4. 错误处理与重试**
- 错误分类：可重试（超时、限流）vs 不可重试（参数错误、权限错误）
- 重试策略：指数退避 + 抖动，设置最大重试次数
- 降级策略：工具不可用时的备选方案
- 幂等性保证：写操作必须带幂等键

**常见追问**：
1. 如何处理工具调用超时？
根据工具调用超时的策略，可以划分为以下几种：
设置合理的超时时间，静态设置超时时间，动态设置超时时间。
异步调用+并行执行：工具异步调用，通过回调或者 event 通知 agent, 没有依赖关系的工具执行可以并发调用，避免拉长链路处理时间。
重试机制：短时间内快速失败并重试，超时或者网络异常快速重试，指数退避，要注意写操作是否有幂等机制。
降级 fallback：降级处理，返回默认结果或者调用其他耗时更短可以替代的工具，例如 websearch 超时，可以改用缓存/记忆工具。
同时还有更细化的策略，分级处理，针对不同工具的重要程度以及下游工具的稳定性/负载，动态的变化处理策略。总的原则可以划分为：重/长任务尽量异步，轻/短任务多次重试，抖动+指数退避避免雪崩。
监控与告警：工具的稳定性以及负载的实时监控与告警。

💡 总结原则：
提前设置合理超时，不要让慢接口拖垮系统。
异步和并行 是处理慢接口的关键。
降级与默认值 保证 Agent 流程可持续运行。
重试+监控 能提高稳定性并及时发现问题。

2. 如何做工具的权限控制？
（结合传统的权限控制以及 agent 权限控制。）
agent 与传统的权限控制结合：RBAC(基于角色的权限控制), ABAC(基于属性/条件的访问控制)
底线控制：工具调用的白名单/黑名单
人机协作：人工确认，敏感操作人工赋权，人工审批。计入一个赋权的临时 key，根据该临时 key 赋予权限。（ 在 QQ 智能体曾经用过的）
工具权限隔离：高风险工具运行与沙箱 sandbox 或者容器中, 微服务通过工具网关调用，由网关负责鉴权。
动态权限与审计：临时授权给 agent，事后审计，必须记录调用的权限，时间，授权人，参数以及返回结果。

分级管理：普通操作、敏感操作、高风险操作 → 不同权限策略。
最小权限原则：Agent 只能访问执行任务必需的工具。
隔离与沙箱：物理或逻辑隔离高风险工具。
动态与审计结合：权限可临时调整，所有操作有审计记录。

3. 如何处理工具返回结果过大的情况？
限制工具的返回量，批量，多次返回，不要一次性返回过多数据。
将工具的全量返回信息进行压缩，将总结后的摘要给到模型，分为文本摘要（LLM 总结）以及结构化抽取（只展示关键信息）两种策略。
结合外部存储+引用机制，工具的结果可以保存到存储介质中，将摘要以及id 返给到模型，模型后续自行判断是是否获取对应信息。
按需加载/懒加载，当工具调用返回多条结果或者结果过大时，可以返回工具调用执行成功的信息，让模型根据需要获取详情，与外部存储+引用机制可结合使用。
分层缓存策略，轻量数据保存在 agent 内部，长期历史缓存，agent 用 key 访问。
返回报错，fallback, 自动降级处理，生成报警日志。

4. 如何做工具调用的可观测性？
针对工具调用的可观测性，可以类比于微服务领域的可观测性。
通过 metric，成功率， trace， log 来观测工具调用，分析。尤其是 agent 调用的 trajectory log（记录了 Agent 在执行任务过程中的完整决策轨迹。）
配置相应的告警与异常处理策略，设定告警阈值，降级策略，异常检测。
可视化面板，成功率，耗时，请求量，监控系统健康程度，资源的按量分配。

**实践示例**

| 场景          | 可观测措施                                | 说明                       |
| ----------- | ------------------------------------ | ------------------------ |
| HTTP API 工具 | Prometheus + Grafana + OpenTelemetry | 记录请求、响应时间、状态码、Trace      |
| 内部任务工具      | Agent 内部日志 + 异常告警                    | 日志记录参数摘要、返回大小、耗时         |
| 高风险操作工具     | 审计日志 + 告警                            | 记录调用者、调用目的、返回状态，异常触发人工通知 |
| 大数据/长任务工具   | 异步追踪 + 分段日志                          | 分块调用可记录每个阶段耗时与结果         |


多维度监控：指标、日志、追踪、告警结合。
链路透明：工具调用过程可追踪，便于快速定位问题。
安全可审计：日志脱敏，异常可追踪。
实时与历史结合：支持实时监控与历史分析优化性能。

## 4. 主干问题：Memory 设计模式有哪些？

### 子问题：Agent 的记忆系统设计

**主问题**：如何设计 Agent 的记忆系统？有哪些类型的记忆？

**口述答案（300-400字）**：

Agent 的记忆系统通常分为三层：

**1. 短期记忆（Short-term Memory）**
- **用途**：存储当前对话的上下文
- **实现**：滑动窗口保留最近 N 轮对话
- **特点**：容量有限，超出后需要压缩或丢弃
- **示例**：最近 10 轮对话历史

**2. 长期记忆（Long-term Memory）**
- **用途**：存储持久化的知识和经验
- **实现**：向量数据库 + 关系数据库
- **特点**：容量大，需要检索才能使用
- **示例**：用户偏好、历史任务、领域知识

**3. 工作记忆（Working Memory）**
- **用途**：存储当前任务的中间状态
- **实现**：结构化的状态对象（JSON/Dict）
- **特点**：任务结束后清空
- **示例**：当前步骤、已完成的子任务、待处理的事项

**记忆压缩策略**：
1. **摘要压缩**：用 LLM 总结历史对话，保留关键信息
2. **重要性过滤**：只保留高价值信息（用户明确要求、关键决策）
3. **分层存储**：热数据在短期记忆，冷数据归档到长期记忆
4. **检索增强**：需要时从长期记忆检索，而不是全部加载

**工程实践**：
- 短期记忆用 Redis/内存，长期记忆用向量库 + 数据库
- 定期做记忆整理（合并、去重、归档）
- 记忆检索时做相关性排序和时效性过滤

**常见追问**：

1. 如何决定哪些信息应该进入长期记忆？

**框架：频率 + 显式 + 跨任务**｜**记忆锚点：高频复现、用户明说、跨任务复用**

判断标准是三条：第一，高频复现——同一信息在多轮或多次会话中反复出现，说明它是持久性偏好而非临时指令。第二，用户显式声明——用户主动说"记住我喜欢 X"或"以后都用 Y"，这是最强信号，必须入库。第三，跨任务可复用——某条信息不只对当前任务有用，未来任务也可能用到（如用户角色、技术栈偏好、业务术语）。反过来，一次性的调试上下文、临时变量、中间推理过程不应进入长期记忆，否则会污染检索质量。工程上可以用一个打分函数（频率 × 显式权重 × 时效衰减）自动判定。

**工程实现**：在每轮对话结束时，用一个轻量级分类器（小模型或规则）判断当前轮是否包含"可持久化信息"。如果包含，提取关键实体（用户偏好、业务术语、操作习惯）写入向量库 + 关系数据库双写：向量库存 embedding 用于语义检索，关系数据库存结构化字段（user_id, key, value, source, timestamp）用于精确查询和统计分析。设置 TTL（如 90 天未使用自动过期），避免冷数据堆积。

2. 记忆压缩会不会丢失重要信息？

**框架：会丢，但可控**｜**记忆锚点：摘要 + 原文索引、分级压缩、人工标记**

会丢，但可以通过三层机制把损失控制在可接受范围。第一层，摘要 + 原文索引：压缩时生成摘要，同时保留原文的存储地址（如向量库 ID），需要时可以回溯原文，摘要负责"快速回忆"，原文负责"精确还原"。第二层，分级压缩：不是所有信息用同一个压缩力度——用户显式指令和关键决策保留原文，普通对话只保留摘要，闲聊直接丢弃。第三层，人工标记：允许用户主动标记"这条很重要"，被标记的信息跳过压缩。本质上压缩是在 token 预算和信息完整性之间做 trade-off，关键是让高价值信息的损失率趋近于零。

**工程实现**：当短期记忆超过阈值（如 20 轮对话），触发压缩流程：用 LLM 生成摘要（prompt: "总结以下对话的关键信息，保留用户偏好、决策和待办事项"），摘要写入短期记忆，原文归档到对象存储（S3/OSS）并记录 archive_id。检索时优先用摘要，如果模型判断需要更多细节，再用 archive_id 拉取原文。分级策略用 importance_score 字段标记（1-5 分），5 分的记忆永不压缩，1-2 分的直接丢弃。

3. 如何处理记忆冲突（新旧信息矛盾）？

**框架：时间戳优先 + 来源可信度 + 冲突标记**｜**记忆锚点：新覆盖旧、来源分级、不确定就标记**

三条规则：第一，时间戳优先——默认新信息覆盖旧信息，因为用户偏好和业务状态会变化，"我现在用 Python"应该覆盖"我之前用 Java"。第二，来源可信度分级——用户显式声明 > Agent 推断 > 模型猜测，低可信度的记忆不能覆盖高可信度的。第三，无法判定时标记冲突而非静默覆盖——把两条矛盾信息都保留，打上 conflict 标签，下次检索到时主动问用户确认，避免 Agent 自作主张选错。工程上在记忆写入时做冲突检测（同一 key 的新旧值比对），触发冲突时走上述规则。

**工程实现**：记忆表增加 source 字段（user_explicit / agent_inferred / model_guessed）和 confidence 字段（0-1）。写入时查询同 key 的最新记录，比较 confidence：新记录 confidence 更高则覆盖，更低则拒绝写入，相等则标记 conflict。冲突记录写入 conflict_log 表，包含 old_value、new_value、conflict_reason。检索时如果命中冲突记录，在返回结果中附加提示："检测到矛盾信息，请确认：A 还是 B？"，用户选择后更新 confidence 并解决冲突。

4. 如何做记忆的版本管理和回滚？

**框架：append-only + 快照 + 软删除**｜**记忆锚点：只追加不覆盖、定期快照、软删除可恢复**

核心思路是 append-only 日志：每次记忆写入不覆盖旧值，而是追加一条新版本记录，带上时间戳和操作类型（create / update / delete）。这样天然支持回滚——回到某个时间点就是只读取该时间点之前的最新版本。定期做快照（如每天一次），快照是当时全量记忆的冻结副本，用于快速恢复而不用重放全部日志。删除用软删除（标记 deleted 而非物理删除），误删可恢复。这套方案本质上和数据库的 MVCC 或 Git 的版本管理是同一个思路，只是粒度是记忆条目而非代码文件。

**工程实现**：记忆表设计为 append-only：每条记录包含 memory_id（不变）、version（递增）、operation（create/update/delete）、content、created_at。查询时用 `SELECT * WHERE memory_id = ? AND operation != 'delete' ORDER BY version DESC LIMIT 1` 拿最新有效版本。回滚到时间点 T：`WHERE created_at <= T`。快照用定时任务（cron）导出当前全量记忆到 JSON 文件存 S3，文件名带时间戳（snapshot_20260427.json）。恢复时直接加载快照文件，比重放日志快 10 倍以上。

## 5. 主干问题：异常恢复策略如何设计？

### 子问题：Agent 的容错与恢复机制

**主问题**：Agent 在生产环境如何处理异常和失败？

**口述答案（300-400字）**：

生产级 Agent 必须有完善的异常恢复机制，分四个层次：

**1. 重试（Retry）**
- **适用场景**：临时性故障（网络抖动、服务限流、超时）
- **策略**：指数退避 + 随机抖动
- **配置**：最大重试次数（3次）、总超时时间（30秒）
- **注意**：写操作必须幂等，避免重复副作用

**2. 降级（Fallback）**
- **适用场景**：依赖服务不可用、工具调用失败
- **策略**：
  - 使用备选工具（主搜索失败 → 备用搜索）
  - 使用缓存结果（实时数据失败 → 历史数据）
  - 简化任务（复杂查询失败 → 简单查询）
- **示例**：向量检索失败 → 降级到关键词检索

**3. 回滚（Rollback）**
- **适用场景**：任务执行到一半发现错误
- **策略**：
  - 保存检查点（Checkpoint），失败时恢复到最近检查点
  - 补偿操作（Compensation），撤销已执行的副作用
  - 事务性操作，要么全成功要么全失败
- **示例**：多步骤任务中某步失败，回滚到上一个稳定状态

**4. 熔断（Circuit Breaker）**
- **适用场景**：下游服务持续失败，避免雪崩
- **策略**：
  - 关闭状态（Closed）：正常调用
  - 打开状态（Open）：直接拒绝，不调用下游
  - 半开状态（Half-Open）：尝试恢复，成功则关闭，失败则继续打开
- **配置**：失败阈值（10次）、超时时间（60秒）、恢复探测间隔（30秒）

**工程实践**：
- 每个工具调用都要有超时和重试配置
- 关键步骤保存检查点，便于恢复
- 全链路 trace_id，便于定位问题
- 实时监控失败率和重试率，异常时告警

**常见追问**：
1. 如何判断一个错误是否可重试？
2. 重试和降级的优先级如何确定？
3. 如何避免重试风暴（所有请求同时重试）？
4. 如何测试异常恢复逻辑？

**优势**：

## 7. 补充原问：AI Agent 的基本实现路径是什么？

### 子问题：如何通过 LLM API、RAG、MCP 组合实现 Agent

**口述答案（约300字）**：
我会把这题回答成"控制面+数据面"。控制面是状态机/图编排：定义节点、路由条件、重试策略和结束条件；数据面是每个节点具体怎么跑。大模型API负责推理与决策，RAG负责提供外部知识，MCP负责标准化工具调用。一个典型回路是：先由模型判断当前任务是否需要知识检索或调用工具；若需要，就先走RAG拿上下文，再走MCP调用外部能力，比如搜索、数据库查询、系统操作；工具结果回填到状态，再交给模型继续推理，直到满足结束条件。关键是"节点输入输出契约"要清晰，比如每个节点只读必要状态，只写自己产物，避免上下文污染。为了线上稳定，我会加步数上限、超时、幂等键和降级策略；为了可运维，我会打全链路trace，区分是Prompt问题、检索问题还是工具问题。

### 展开：控制面 + 数据面详解

**一、控制面（Control Plane）—— 决定"做什么、怎么走、何时停"**

控制面是 Agent 的骨架，负责编排整个任务的执行流程。核心职责是三件事：路由决策、状态管理、异常治理。

**1. 路由决策**
- 每一轮循环，控制面根据当前状态决定下一步走哪个节点：直接回复、走 RAG 检索、还是走工具调用
- 实现方式有两种：
  - **模型驱动路由**：把路由决策交给 LLM，模型输出 next_action 字段（如 `"action": "search"`），控制面解析后分发
  - **规则驱动路由**：用条件判断（if/switch）硬编码路由逻辑，适合流程确定的场景，稳定性更高
- 生产建议：核心链路用规则路由保底，开放性任务用模型路由，两者可以混合

**2. 状态管理**
- 状态对象是控制面的核心数据结构，在节点间传递，记录任务全貌：
  ```json
  {
    "task_id": "xxx",
    "current_step": 2,
    "max_steps": 10,
    "history": [...],
    "tool_results": {...},
    "retrieval_context": "...",
    "status": "running"
  }
  ```
- 每个节点只读自己需要的字段，只写自己的产物，避免上下文污染
- 关键检查点（Checkpoint）持久化到存储，失败时可从最近检查点恢复，不用从头跑

**3. 异常治理与终止条件**
- 步数上限：防止死循环（如最多 10 轮）
- 总超时：防止慢任务拖垮系统
- 熔断：某个工具连续失败 N 次后跳过，走降级路径
- 结束条件：模型输出 `"done": true` / 达到步数上限 / 超时 / 人工中断

**二、数据面（Data Plane）—— 决定"每个节点具体怎么跑"**

数据面是控制面调度的执行单元，每个节点是一个原子能力。主要有三类节点：

**1. LLM 推理节点**
- 职责：理解意图、生成回复、决策下一步动作
- 输入：system prompt + 当前状态（对话历史、检索结果、工具返回）
- 输出：回复文本 + 结构化决策（是否需要工具/检索、调用哪个工具、参数是什么）
- 工程要点：prompt 模板化管理，不同任务类型用不同 prompt；token 预算控制，避免上下文超窗口

**2. RAG 检索节点**
- 职责：从外部知识库召回与当前问题相关的文档片段，补充模型不知道的信息
- 流程：Query 改写 → 向量检索 + 关键词检索（混合召回）→ Rerank 精排 → 截断拼入 context
- 输入：用户问题或模型改写后的检索 query
- 输出：top-K 文档片段，附带来源引用
- 工程要点：分块策略影响召回质量；检索结果要控制长度，避免挤占 LLM 推理的 token 空间

**3. 工具调用节点（MCP）**
- 职责：执行外部操作（搜索、数据库查询、API 调用、系统操作）
- MCP 的价值：标准化工具接口协议，不同工具用统一的 JSON Schema 注册、统一的请求/响应格式，Agent 不需要为每个工具写适配代码
- 流程：模型输出工具名 + 参数 → 控制面分发到工具网关 → 参数校验 → 执行 → 结果标准化 → 回填状态
- 输入：工具名、参数（由 LLM 生成）
- 输出：标准化结果 `{code, data, message, retryable}`
- 工程要点：权限控制（最小权限）、幂等保证（写操作带幂等键）、超时重试、降级备选

**三、控制面与数据面的协作回路**

```
用户输入 → 控制面：初始化状态，进入循环
              ↓
         LLM 推理节点：分析任务，输出决策
              ↓
         控制面：解析决策，路由到下一个节点
           ├→ 需要知识 → RAG 检索节点 → 结果回填状态 → 回到 LLM
           ├→ 需要执行 → MCP 工具节点 → 结果回填状态 → 回到 LLM
           └→ 任务完成 → 返回最终结果
              ↓
         控制面：检查终止条件（步数/超时/完成标志）
           ├→ 未满足 → 继续循环
           └→ 已满足 → 结束，输出结果
```

**四、线上治理（跨控制面和数据面）**

- 全链路 trace_id：串联每一轮的 LLM 调用、检索、工具执行，出问题时快速定位是 Prompt 问题、检索问题还是工具问题
- 成本控制：token 用量监控、热点查询缓存、简单任务路由到小模型
- 灰度发布：新 prompt / 新工具先小流量验证，再全量放开

**面试表达要点**：
- 先说"控制面+数据面"的分法，再分别展开，逻辑清晰
- 控制面重点讲路由、状态、异常三件事
- 数据面重点讲三类节点各自的输入输出和工程要点
- 最后用协作回路串起来，说明它们怎么配合工作
- 提到治理和 trace，说明你考虑的是生产级而不是 demo 级

## 9. 补充原问：如果设计一个 Agent 产品，你会从什么方向切入？

### 子问题：Agent 产品设计思路

**口述答案（约300字）**：
我会优先选"高频、可量化、可回滚"的业务场景，比如企业知识问答+任务执行助手。设计思路是先确定目标指标，再反推功能。第一步定义北极星指标，如任务完成率、人工介入率、平均处理时长、单位任务成本。第二步做能力拆分：意图识别、检索增强、工具执行、结果审校、异常兜底。第三步做安全和治理：最小权限、敏感操作二次确认、全链路审计。第四步做上线策略：小流量灰度、在线AB、失败快速回滚。我的经验是先做"半自动"而不是"全自动"，把高风险动作放人工审批，先把稳定性和信任建立起来，再逐步放开自治程度。这样产品落地速度更快，业务方也更愿意采用。

### 展开：从场景选择到上线落地

**一、场景选择 —— 为什么选"高频、可量化、可回滚"**

不是所有场景都适合上 Agent，选错场景比技术做不好更致命。筛选标准：

| 维度 | 适合 Agent | 不适合 Agent |
|------|-----------|-------------|
| 频率 | 高频重复（每天几百上千次） | 低频一次性任务 |
| 可量化 | 有明确的成功/失败判定标准 | 结果好坏靠主观感受 |
| 可回滚 | 操作可撤销或影响可控 | 不可逆操作（如资金转账、删除数据） |
| 容错空间 | 偶尔出错可接受（如推荐、摘要） | 零容错（如医疗诊断、法律判决） |
| 人力成本 | 当前人工处理成本高 | 人工处理已经很高效 |

好的切入场景举例：
- 企业内部知识问答 + 工单处理：高频、可量化（解决率）、出错可人工兜底
- 内容生产辅助（摘要、改写、审校）：高频、结果可评分、不涉及不可逆操作
- 客服分流 + 自动回复：高频、可量化（转人工率）、回答错了用户可以追问

不好的切入场景：
- 全自动交易执行（不可回滚、零容错）
- 低频的一次性分析报告（ROI 不够）

**二、指标定义 —— 北极星指标 + 过程指标**

先定指标再做功能，避免"做了很多但不知道好不好"。

北极星指标（衡量产品价值）：
- **任务完成率**：Agent 独立完成任务的比例（不需要人工介入）
- **人工介入率**：需要转人工或人工修正的比例（越低越好）
- **单位任务成本**：完成一个任务的平均 token + API 调用成本

过程指标（定位问题）：
- **意图识别准确率**：用户意图是否被正确理解
- **检索召回率 / 准确率**：RAG 是否找到了正确的知识
- **工具调用成功率**：工具是否正常执行
- **平均处理时长**：从用户输入到最终回复的端到端耗时
- **轮次分布**：完成任务平均需要几轮交互（越少越好）

指标之间的因果关系：
```
意图识别准确率 ↑ → 检索/工具选择更准 → 任务完成率 ↑ → 人工介入率 ↓
工具调用成功率 ↑ → 重试减少 → 平均耗时 ↓ + 成本 ↓
```

**三、能力拆分 —— 五个模块逐步搭建**

不要一步到位做全自动，按模块拆分，逐个验证：

1. **意图识别**：判断用户要做什么（查询、操作、闲聊），决定走哪条链路
2. **检索增强（RAG）**：从知识库召回相关信息，补充模型不知道的领域知识
3. **工具执行**：调用外部系统完成操作（查数据、提工单、发通知）
4. **结果审校**：对 Agent 的输出做二次检查（格式校验、事实核查、安全过滤）
5. **异常兜底**：工具失败降级、模型幻觉检测、超时熔断、转人工

搭建顺序建议：先做 1+2（能回答问题），再加 3（能做事），再加 4+5（能可靠地做事）。

**四、安全与治理 —— 先建信任再放权**

Agent 产品最大的阻力不是技术，是信任。业务方不敢用，因为怕出事。

信任建设策略：
- **最小权限**：Agent 只能访问完成任务必需的工具和数据，不给多余权限
- **分级操作**：
  - 只读操作（查询、搜索）→ 自动执行
  - 低风险写操作（创建草稿、添加标签）→ 自动执行 + 事后审计
  - 高风险写操作（发布、删除、修改权限）→ 人工确认后执行
- **全链路审计**：每次操作记录 who（哪个 Agent）、what（做了什么）、why（基于什么决策）、when（什么时候）
- **输入输出过滤**：防 prompt injection、过滤敏感信息、检测幻觉

**五、上线策略 —— 从半自动到全自动**

```
阶段 1：人工主导，Agent 辅助
  → Agent 给建议，人工决定是否采纳
  → 目标：验证 Agent 建议的准确率

阶段 2：Agent 主导，人工审批
  → Agent 自动执行，高风险操作需人工确认
  → 目标：验证端到端任务完成率

阶段 3：Agent 自动执行，人工抽检
  → Agent 全自动，人工按比例抽查
  → 目标：验证长期稳定性和成本

阶段 4：全自动 + 异常告警
  → 只在异常时通知人工
  → 目标：持续运行，持续优化
```

每个阶段的晋升条件：
- 阶段 1 → 2：建议采纳率 > 80%
- 阶段 2 → 3：任务完成率 > 90%，人工修正率 < 10%
- 阶段 3 → 4：连续 N 天无严重事故，抽检合格率 > 95%

**面试表达要点**：
- 先说场景选择标准（高频、可量化、可回滚），体现产品判断力
- 再说指标驱动（先定指标再做功能），体现数据思维
- 重点讲"半自动到全自动"的渐进策略，这是最能体现工程落地经验的部分
- 如果有实际项目，在每个阶段举一个你经历过的具体例子
