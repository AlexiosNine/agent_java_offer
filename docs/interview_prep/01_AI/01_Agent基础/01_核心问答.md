## 1. 主干问题：什么是基于 LLM 的 Agent，它和普通大模型有什么区别？

### 子问题：Agent 定义与核心组件

**主问题**：你如何定义基于 LLM 的 Agent？核心组件有哪些？

**口述答案**：
我会把 Agent 定义为"以 LLM 为决策中枢、可在环境中闭环执行任务的系统"。它通常由四层构成：

1. **规划层（Planning）**：任务分解、步骤规划、目标设定
2. **记忆层（Memory）**：短期记忆（对话历史）、长期记忆（知识库）、工作记忆（任务状态）
3. **工具层（Tools）**：外部能力调用（搜索、数据库、API）
4. **执行控制层（Control）**：状态管理、异常恢复、可观测性

和普通对话模型相比，关键差异是它能持续感知、决策、行动并根据反馈修正。若从软件 Agent 走向具身或仿真 Agent，难点会从"文本理解"升级到"高噪声感知 + 连续动作控制 + 安全约束"，系统设计重点也会从回答质量转为"安全、稳定、可恢复"。

**常见追问**：
1. Agent 与 Workflow 最大差别是什么？
2. 为什么说"可观测性"比"会答题"更重要？
3. 具身场景为什么更强调实时性与安全兜底？

## 2. 主干问题：ReAct 是什么，Agent 的规划能力怎么设计？

### 子问题：ReAct 框架详解

**主问题**：请详细解释 ReAct 框架。它是如何将思维链和行动结合起来的？

**口述答案**：
ReAct 的本质是把"推理"和"行动"交替执行，形成 **Thought-Action-Observation** 的循环：

1. **Thought（思考）**：模型分析当前状态，决定下一步做什么
2. **Action（行动）**：调用工具执行具体动作
3. **Observation（观察）**：获取工具返回结果，更新状态
4. 循环直到任务完成

它相比静态 CoT 的优势是可交互、可纠错、可解释，适合信息不完整或环境会变化的任务；代价是链路更长，线上要控制步数、超时和成本。

**规划能力对比**：
- **CoT（Chain of Thought）**：单路径分步推理，成本低但容错弱
- **ToT（Tree of Thoughts）**：树状分支探索和回溯，成功率高但算力开销大
- **GoT（Graph of Thoughts）**：图结构允许分支合并和循环优化，适合复杂依赖

**工程实践**：建议把"规划"和"执行"解耦：Planner 负责拆解，Executor 负责落地，减少模型一步到位失败。

**常见追问**：
1. 什么时候不该用复杂规划而该用固定流程？
2. 规划深度如何控制成本？
3. 如何避免"想太多、做太少"？

## 3. 主干问题：Tool Calling 机制如何设计？

### 子问题：Tool Calling 完整流程

**主问题**：如何设计一个生产级的 Tool Calling 系统？

**口述答案（300-400字）**：

Tool Calling 的核心是"模型决策 + 工具网关执行"，分四个阶段：

**1. 函数定义与注册**
- 用 JSON Schema 定义工具签名（名称、描述、参数、返回值）
- 注册到工具注册表，支持版本管理和动态加载
- 示例：
```json
{
  "name": "search",
  "description": "搜索相关信息",
  "parameters": {
    "query": {"type": "string", "description": "搜索关键词"}
  }
}
```

**2. 模型调用与参数解析**
- 模型根据任务决定调用哪个工具
- 生成结构化的函数调用请求（function name + arguments）
- 解析模型输出，提取工具名和参数（JSON 解析或正则匹配）

**3. 工具执行与结果处理**
- 通过工具网关执行（不是模型直接调用）
- 参数校验：类型检查、范围检查、权限检查
- 执行工具并捕获异常
- 标准化返回格式：`{code, message, data, retryable, trace_id}`

**4. 错误处理与重试**
- 错误分类：可重试（超时、限流）vs 不可重试（参数错误、权限错误）
- 重试策略：指数退避 + 抖动，设置最大重试次数
- 降级策略：工具不可用时的备选方案
- 幂等性保证：写操作必须带幂等键

**常见追问**：
1. 如何处理工具调用超时？
2. 如何做工具的权限控制？
3. 如何处理工具返回结果过大的情况？
4. 如何做工具调用的可观测性？

## 4. 主干问题：Memory 设计模式有哪些？

### 子问题：Agent 的记忆系统设计

**主问题**：如何设计 Agent 的记忆系统？有哪些类型的记忆？

**口述答案（300-400字）**：

Agent 的记忆系统通常分为三层：

**1. 短期记忆（Short-term Memory）**
- **用途**：存储当前对话的上下文
- **实现**：滑动窗口保留最近 N 轮对话
- **特点**：容量有限，超出后需要压缩或丢弃
- **示例**：最近 10 轮对话历史

**2. 长期记忆（Long-term Memory）**
- **用途**：存储持久化的知识和经验
- **实现**：向量数据库 + 关系数据库
- **特点**：容量大，需要检索才能使用
- **示例**：用户偏好、历史任务、领域知识

**3. 工作记忆（Working Memory）**
- **用途**：存储当前任务的中间状态
- **实现**：结构化的状态对象（JSON/Dict）
- **特点**：任务结束后清空
- **示例**：当前步骤、已完成的子任务、待处理的事项

**记忆压缩策略**：
1. **摘要压缩**：用 LLM 总结历史对话，保留关键信息
2. **重要性过滤**：只保留高价值信息（用户明确要求、关键决策）
3. **分层存储**：热数据在短期记忆，冷数据归档到长期记忆
4. **检索增强**：需要时从长期记忆检索，而不是全部加载

**工程实践**：
- 短期记忆用 Redis/内存，长期记忆用向量库 + 数据库
- 定期做记忆整理（合并、去重、归档）
- 记忆检索时做相关性排序和时效性过滤

**常见追问**：
1. 如何决定哪些信息应该进入长期记忆？
2. 记忆压缩会不会丢失重要信息？
3. 如何处理记忆冲突（新旧信息矛盾）？
4. 如何做记忆的版本管理和回滚？

## 5. 主干问题：异常恢复策略如何设计？

### 子问题：Agent 的容错与恢复机制

**主问题**：Agent 在生产环境如何处理异常和失败？

**口述答案（300-400字）**：

生产级 Agent 必须有完善的异常恢复机制，分四个层次：

**1. 重试（Retry）**
- **适用场景**：临时性故障（网络抖动、服务限流、超时）
- **策略**：指数退避 + 随机抖动
- **配置**：最大重试次数（3次）、总超时时间（30秒）
- **注意**：写操作必须幂等，避免重复副作用

**2. 降级（Fallback）**
- **适用场景**：依赖服务不可用、工具调用失败
- **策略**：
  - 使用备选工具（主搜索失败 → 备用搜索）
  - 使用缓存结果（实时数据失败 → 历史数据）
  - 简化任务（复杂查询失败 → 简单查询）
- **示例**：向量检索失败 → 降级到关键词检索

**3. 回滚（Rollback）**
- **适用场景**：任务执行到一半发现错误
- **策略**：
  - 保存检查点（Checkpoint），失败时恢复到最近检查点
  - 补偿操作（Compensation），撤销已执行的副作用
  - 事务性操作，要么全成功要么全失败
- **示例**：多步骤任务中某步失败，回滚到上一个稳定状态

**4. 熔断（Circuit Breaker）**
- **适用场景**：下游服务持续失败，避免雪崩
- **策略**：
  - 关闭状态（Closed）：正常调用
  - 打开状态（Open）：直接拒绝，不调用下游
  - 半开状态（Half-Open）：尝试恢复，成功则关闭，失败则继续打开
- **配置**：失败阈值（10次）、超时时间（60秒）、恢复探测间隔（30秒）

**工程实践**：
- 每个工具调用都要有超时和重试配置
- 关键步骤保存检查点，便于恢复
- 全链路 trace_id，便于定位问题
- 实时监控失败率和重试率，异常时告警

**常见追问**：
1. 如何判断一个错误是否可重试？
2. 重试和降级的优先级如何确定？
3. 如何避免重试风暴（所有请求同时重试）？
4. 如何测试异常恢复逻辑？

## 6. 主干问题：Agent 框架如何选型？

### 子问题：LangGraph vs CrewAI vs AutoGen 对比

**主问题**：主流 Agent 框架有哪些？各自的优劣势是什么？

**口述答案（300-400字）**：

**1. LangGraph（LangChain 生态）**

**优势**：
- 图结构编排，支持复杂的条件分支和循环
- 状态管理清晰，每个节点输入输出明确
- 与 LangChain 生态无缝集成（LLM、工具、向量库）
- 支持流式输出和中间状态查看
- 文档完善，社区活跃

**劣势**：
- 学习曲线陡峭，概念较多（State、Node、Edge、Conditional Edge）
- 代码量较大，简单任务也需要较多配置
- 性能开销相对较高

**适用场景**：复杂的多步骤任务、需要条件分支和循环的场景

**2. CrewAI**

**优势**：
- 多 Agent 协作模式，角色分工清晰（Manager、Worker、Reviewer）
- 声明式配置，代码简洁
- 内置任务分配和结果汇总机制
- 适合团队协作场景

**劣势**：
- 灵活性较低，不适合高度定制化的场景
- 调试困难，Agent 之间的通信不够透明
- 社区相对较小

**适用场景**：多角色协作任务、需要任务分配和汇总的场景

**3. AutoGen（Microsoft）**

**优势**：
- 对话式 Agent 交互，自然语言驱动
- 支持人机协作（Human-in-the-loop）
- 代码执行能力强，适合编程任务
- 支持多模型协作（不同 Agent 用不同模型）

**劣势**：
- 控制流不够清晰，依赖对话驱动容易失控
- 成本较高（多轮对话消耗 token 多）
- 可观测性较弱

**适用场景**：编程任务、需要人机协作的场景、多模型协作

**选型建议**：
- 复杂流程控制 → LangGraph
- 多角色协作 → CrewAI
- 编程任务 + 人机协作 → AutoGen
- 简单任务 → 自己实现（100-200 行代码）

**常见追问**：
1. 这些框架的性能对比如何？
2. 如何从一个框架迁移到另一个框架？
3. 是否可以混合使用多个框架？
4. 生产环境推荐哪个框架？

## 7. 补充原问：AI Agent 的基本实现路径是什么？

### 子问题：如何通过 LLM API、RAG、MCP 组合实现 Agent

**口述答案（约300字）**：
我会把这题回答成"控制面+数据面"。控制面是状态机/图编排：定义节点、路由条件、重试策略和结束条件；数据面是每个节点具体怎么跑。大模型API负责推理与决策，RAG负责提供外部知识，MCP负责标准化工具调用。一个典型回路是：先由模型判断当前任务是否需要知识检索或调用工具；若需要，就先走RAG拿上下文，再走MCP调用外部能力，比如搜索、数据库查询、系统操作；工具结果回填到状态，再交给模型继续推理，直到满足结束条件。关键是"节点输入输出契约"要清晰，比如每个节点只读必要状态，只写自己产物，避免上下文污染。为了线上稳定，我会加步数上限、超时、幂等键和降级策略；为了可运维，我会打全链路trace，区分是Prompt问题、检索问题还是工具问题。

## 8. 补充原问：Agent 当前的局限是什么？

### 子问题：Agent 的缺点和挑战

**口述答案（约300字）**：
我认为Agent当前的短板主要是"稳定性、可控性、成本"三件事。第一，稳定性：任务链路长、依赖外部工具多，任何一环抖动都会影响成功率。第二，可控性：模型是概率系统，复杂场景下会出现误判、过度调用工具甚至越权风险。第三，成本：多轮推理+检索+工具调用叠加，token和外部API成本上升很快。为什么看起来"没爆款"，本质是很多场景还没跨过"可用到可靠"的工程门槛。我的看法是Agent更适合流程明确、容错可设计、收益可量化的场景，比如内容生产、客服分流、内部效率工具。要做出亮点，关键不是堆模型，而是做强编排、评测和治理，把失败率和成本打下来。

## 9. 补充原问：如果设计一个 Agent 产品，你会从什么方向切入？

### 子问题：Agent 产品设计思路

**口述答案（约300字）**：
我会优先选"高频、可量化、可回滚"的业务场景，比如企业知识问答+任务执行助手。设计思路是先确定目标指标，再反推功能。第一步定义北极星指标，如任务完成率、人工介入率、平均处理时长、单位任务成本。第二步做能力拆分：意图识别、检索增强、工具执行、结果审校、异常兜底。第三步做安全和治理：最小权限、敏感操作二次确认、全链路审计。第四步做上线策略：小流量灰度、在线AB、失败快速回滚。我的经验是先做"半自动"而不是"全自动"，把高风险动作放人工审批，先把稳定性和信任建立起来，再逐步放开自治程度。这样产品落地速度更快，业务方也更愿意采用。
