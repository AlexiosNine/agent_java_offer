## 三、核心短板（改进）

### 短板 1：题目要求的必须回答 ⭐⭐⭐⭐⭐

**问题描述**：
题目明确要求的部分经常遗漏或一笔带过

**典型案例**：
- 第 4 题：题目问"怎么分阶段落地"，但回答只讲了指标，完全没讲落地路径
- 第 5 题：题目要求"结合具体场景"，但回答没有给出任何场景
- 第 6 题：题目核心问"如何定位问题出在 Prompt、RAG 还是工具"，但只用一句话带过

**扣分统计**：
- 第 4 题：-1.5 分
- 第 5 题：-1.5 分
- 第 6 题：-3 分
- **累计扣分：6 分**（占总扣分的 40%）

**根本原因**：
1. 听题不仔细，没有识别出题目的多个子问题
2. 没有判断哪个子问题是重点
3. 时间分配不合理，把大量时间花在非重点部分

**改进方法**：
```
听题 → 拆题 → 判断权重 → 分配时间

例如第 6 题：
- 子问题 1：离线评估（20%）
- 子问题 2：在线评估（20%）
- 子问题 3：问题定位（60%，重点！）

时间分配：
- 离线评估：30 秒
- 在线评估：30 秒
- 问题定位：1.5-2 分钟
```

**练习方法**：
- 拿到题目后，先在纸上列出子问题
- 标注每个子问题的权重（10% / 20% / 50%）
- 按权重分配回答时间
- 重点部分必须占 50% 以上篇幅

### 短板 2：要有具体数字，不要模糊表达 ⭐⭐⭐⭐

**问题描述**：
经常用"一定程度"、"达到阈值"等模糊表达，缺少具体数字

**典型案例**：
- 第 5 题："上下文达到一定程度" → 应该说"20 轮对话或 8000 tokens"
- 第 1 题："N 轮对话" → 应该说"最近 10 轮"

**扣分统计**：
- 第 5 题：-1 分
- 第 1 题：-0.5 分
- **累计扣分：1.5 分**

**根本原因**：
1. 担心说错具体数字
2. 没有准备常见的阈值参考值

**改进方法**：

**常见阈值参考表**：

| 场景 | 参数 | 推荐值 | 说明 |
|------|------|--------|------|
| 短期记忆 | 轮次 | 10-20 轮 | 超过触发压缩 |
| 短期记忆 | Token | 8000-16000 | 接近模型窗口的 50% |
| 工具超时 | 轻量任务 | 5 秒 | 查询、搜索 |
| 工具超时 | 重量任务 | 30-60 秒 | 大数据分析 |
| 重试次数 | 最大重试 | 3 次 | 避免无限重试 |
| 重试间隔 | 指数退避 | 1s → 2s → 4s | 第 n 次等 2^(n-1) 秒 |
| 熔断阈值 | 连续失败 | 10 次 | 触发熔断 |
| 熔断恢复 | 半开时间 | 60 秒 | 尝试恢复 |
| 记忆 TTL | 冷数据过期 | 90 天 | 未命中自动降权 |
| 灰度比例 | 初始流量 | 5% | 逐步放量到 100% |

**记忆技巧**：
- 短期记忆：10-20 轮，8K tokens
- 超时：轻 5 秒，重 30 秒
- 重试：最多 3 次，指数退避
- 熔断：连续失败 10 次，60 秒恢复

### 短板 3：缺少具体场景和例子 ⭐⭐⭐⭐

**问题描述**：
回答偏理论，缺少具体场景展示完整流程

**典型案例**：
- 第 2 题：ReAct 的 Coding 场景描述太笼统，没有展示每一轮的输入输出
- 第 5 题：完全没有给场景

**扣分统计**：
- 第 2 题：-1 分
- 第 5 题：-1.5 分
- **累计扣分：2.5 分**

**改进方法**：

**场景模板**：
```
场景：[具体业务场景]
目标：[要完成的任务]

流程：
- 轮 1：Thought: [分析] → Action: [工具调用] → Obs: [结果]
- 轮 2：Thought: [分析] → Action: [工具调用] → Obs: [结果]
- 轮 3：Thought: [分析] → Action: [工具调用] → Obs: [结果]
- 轮 N：Thought: [任务完成] → Action: Finish

关键点：[每轮的 Observation 如何影响下一轮的 Thought]
```

**常备场景库**：
1. **客服 Agent**：用户咨询订单 → 查询订单 → 查物流 → 推送信息
2. **天气 Agent**：查天气 → 判断下雨 → 查日历 → 改约会
3. **Coding Agent**：读代码 → 理解需求 → 写代码 → 测试 → 修复
4. **搜索 Agent**：理解问题 → 搜索 → 提取答案 → 验证 → 返回

### 短板 4：用决策树而非清单 ⭐⭐⭐

**问题描述**：
回答是"平铺"的策略列举，没有形成清晰的决策流程

**典型案例**：
- 第 3 题：列举了重试、降级、熔断、转人工，但没有说"什么情况下走哪条路径"

**扣分统计**：
- 第 3 题：-1 分

**改进方法**：

**决策树模板**：
```
第一步：判断 [条件 A]
  ├─ 如果 [Y]：走策略 A
  │   └─ 策略 A 失败 → 走策略 B
  └─ 如果 [N]：走策略 C
      └─ 策略 C 失败 → 走策略 D

终止条件：[成功 / 超时 / 达到最大重试次数]
```

**示例（工具超时处理）**：
```
第一步：判断任务类型
  ├─ 轻量任务（<5秒）
  │   ├─ 重试 3 次（指数退避）
  │   ├─ 重试失败 → 降级（缓存/默认值）
  │   └─ 降级失败 → 返回错误
  └─ 重量任务（>5秒）
      ├─ 异步执行 + 回调
      ├─ 超时 → 熔断（连续失败 10 次）
      └─ 关键路径 → 转人工
```

---

### 短板 5：缺少可观测性和监控 ⭐⭐⭐

**问题描述**：
只讲"怎么处理"，不讲"怎么发现问题"和"怎么定位根因"

**典型案例**：
- 第 3 题：讲了超时处理策略，但没讲如何监控超时率、如何定位是哪个环节超时
- 第 6 题：提到了 trace，但没说具体怎么用 trace 定位问题

**扣分统计**：
- 第 3 题：-0.5 分
- 第 6 题：-3 分（核心问题未回答）
- **累计扣分：3.5 分**

**改进方法**：

**可观测性三件套**：
1. **Metrics（指标）**：超时率、重试率、降级率、熔断触发次数
2. **Trace（链路追踪）**：trace_id 串联 LLM 推理 → 检索 → 工具调用
3. **Log（日志）**：trajectory log 记录每轮的 Thought → Action → Observation

**问题定位方法（控制变量法）**：
```
发现问题 → 看 trace 定位失败环节 → 隔离测试确认根因

Prompt 问题：
  - 固定检索结果和工具，换 Prompt
  - 输出变好 → Prompt 问题

RAG 问题：
  - 用 golden context 替换检索结果
  - 输出变好 → 检索问题
  - 查召回率、精确率

工具问题：
  - 用 mock 工具结果替换
  - 输出变好 → 工具问题
  - 查工具成功率、超时率
```
