

## 1. mcp, skill 和 rag 解释下?

Skill 更像“方法论”或者“流程模板”, 更适合封装重复性流程和团队经验，它定义的是模型遇到某类任务时应该按什么步骤做；
RAG 解决的是“知识从哪里来”的问题，它把当前任务真正相关、而且可能实时变化的外部信息检索出来，再喂给模型。
所以 Skill 是 能力封装，RAG 是 知识供给，两者通常是配合关系，不是替代关系。
甚至很多时候，一个 Skill 内部就会规定：先检索、再筛选、再生成、再校验。这里面的“检索”本身往往就是 RAG。

## 2. Claude Code 的 Subagent 模式和 Agent Team 的模式有什么区别呢？

Subagent: 一般比较会处理一些比较轻量级的，短周期的一些事务。比如说开发某一个用户模块的某一个登录的功能，然后呢把开发这一个功能的时候。把这几个功能拆分为，方案设计、前端开发、后台开发和测试这么几个子的任务。然后由一个主的 Agent 来负责，就是分任务和，并且去和各个 Sub Agent 之间进行通信。然后当这个任务整个开发进行执行完之后，这个 Subway 真的就会被回收，它的周期是比较短的。

Agent Team: 就会负责一个比较大的任务，然后也适适合就是整个上下游，多团队多角色的并行的一个协作。agent的 team 之间呢，他们是是一些共享一些状态和记忆和任务看板的。咱们协作起来有团队的路由、评审、之类投票之类的一些机制。好像我们现实工作中开发一个大项目，比如说涉及到多个链路模块，涉及到用户模块，活动模块，然后通知模块等涉及到多个团队的并行，就是像多个团队这样协作的沟通。



## 3. 什么是 Harness Engineering？​

"Harness"直译是马具, 中文可以叫:驾驭工程, 它的核心思想是：系统性地设计约束、反馈链路和持续改进机制，使 AI Agent 能够在大规模、长周期的软件开发中可靠运行。它包含系统提示词、工具、文件系统、沙箱、编排逻辑和检查机制，本质上回答三个问题：AI 在哪干活？用什么干活？怎么知道干得对不对？

OpenAI 的思路是搭好工作台。  
用精简的目录文件加结构化文档体系 和工具信息，让 AI 在需要时找到对应信息 (渐进式披露)。再给 AI 配备了完整的工具链和可观测性堆栈——日志、指标、UI 界面全部暴露给 AI，让它能看到代码运行的真实状态，并重写了 Linter(静态代码检查工具) 错误信息，使其面向 AI 而非人类, 使得AI 可以根据反馈进行持续的改进。

我自己实践下来就是，除了有一个开发的一个 Agent ，还会一定要并行起一个测试的 Agent 。把反馈链路给打通, 让这个测试 Agent 它能够读到所有的那个调用的工具链的一个可观测性的日志，然后呢并且让这个测试 Agent 它能够端到端的，就是去去模仿人在浏览器上去进行每个 功能的一个验证。然后呢自己去收集报多信息，然后给到开发去来进行修正。

使其面向 AI 而非人类:
OpenAI 重写 Linter 错误信息的目标，不是单纯的说哪里错了, 是让每一条报错都包含 AI 做决策所需的完整信息，大致包括：
规则的意图：这条规则存在是为了防止什么问题？
具体的修复方向：不是"你错了"，而是"你应该这样改"。
上下文约束：比如项目的架构规则要求依赖方向是单向的，违反时要说清楚是哪条链路出了问题。

Harness Engineering 的本质是把你的工作方法和判断标准，转化为 AI 能理解和执行的环境。每当 AI 出错，应该反问"我的环境里缺了什么"——这是工程师角色从"写代码"转向"设计环境/标准、建立反馈"的根本转变。

## 4. claude code源码泄漏,cc的架构设计的启发

从 Claude Code 这次源码意外曝光里，作为 Agent 开发者，我们到底能学到什么？

---

### **第一课：agent的工程特别重要，不不光光全是模型的能力，工程设计也很重要。**

Claude Code 的源码——51万行代码里，真正调用 LLM API 的部分不到 5%。剩下那 95% 是什么？是安全检查、权限管理、上下文压缩、错误恢复、多 Agent 协调。

换句话说，模型只是发动机，你造的是整辆车。发动机再好，没有底盘、没有刹车、没有方向盘，车照样跑不起来。

---

### **第二课：提示词分层管理**

Claude Code 的做法完全不同：它把提示词拆成静态层和动态层，静态部分走缓存节省费用，动态部分每次注入实时环境信息；每个工具还配一份专门给 AI 看的"使用手册"，明确告诉模型这个工具能干什么、绝对不能干什么。

---

### **第三课：安全的重视**

Claude Code 为了让一个 BashTool 能安全运行，写了 18 个安全相关文件，设计了 9 层审查流程。他们默认所有工具状态是"不安全的、会写入的"，除非开发者明确声明它是安全的，否则系统就按最保守的方式处理。

对我们来说，**在设计工具权限的时候，默认收紧，按需放开，而不是默认放开，出事再收**。尤其是涉及文件读写、网络请求、数据库操作这类有副作用的工具，一定要在设计阶段就想清楚边界在哪里。

---

### **第四课：上下文管理是 Agent 的生命线，必须主动设计**

上下文窗口 三层递进压缩：先只清理旧的工具调用结果（对话主线保留），再在 token 用量接近 87% 时自动触发压缩，最后实在不行了才让 AI 生成全局摘要替换历史。

这个思路对我们很有参考价值：**上下文管理不是一个开关，而是一套策略**。你需要分清楚哪些内容是"可以丢的"（比如中间过程的工具返回结果），哪些是"必须保留的"（比如用户的核心指令和对话主线），然后按优先级来处理。

---

### **第五课：记忆系统要"精确"，不要"贪多"**

Claude Code 的记忆检索策略有一句核心原则：精确度优先于召回率。意思是，宁可漏掉一条可能有用的记忆，也不把不相关的记忆塞进上下文污染模型的判断。

我们通常觉得"多给点信息总没错"。但对 LLM 来说，无关信息不是中性的，它会干扰模型的注意力，让输出质量下降。所以在设计记忆系统时，与其堆砌所有历史，不如想清楚：**这条记忆对当前任务真的有帮助吗？** 如果不确定，就不要注入。

---

### **第六课：多 Agent 协作要给每个 Agent 明确的"身份"**

Claude Code 在生成子 Agent 时，会往它的上下文里注入一段非常强硬的指令：你是一个工人，不是经理，不许再派活给别人，直接干，汇报结果不超过500字。防止这个子agent再无限生成它的子agent。

对我们的启示是：**在多 Agent 架构里，每个 Agent 的职责边界必须在设计阶段就写死，而不是靠它自己"聪明地"判断**。协调者就只协调，执行者就只执行.

---

围绕它的安全机制、上下文管理、记忆系统、多 Agent 协调、提示词工程——这些"配套设施"才是决定产品能不能真正好用的关键。Claude Code 用 51 万行代码告诉了我们这个答案，而我们不需要重走一遍，直接拿来用就好。
